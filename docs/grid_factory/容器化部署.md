# GridFactory 容器化部署

本文件说明如何以容器化方式运行 GridFactory 算法引擎（`sky_executor.grid_factory`）。
算法侧的参考环境位于独立仓库 `SkyEngine`（分支 `0602stable` 及以后），提供 `sim_server.py` 在线控制服务、`online.dockerfile` 镜像构建，以及两套 compose 编排文件。

> 如果你要在 Application（FastAPI + Vue）里通过 `joint_sim` 集成 GridFactory，请看本目录的 [QUICKSTART.md](./QUICKSTART.md)。本文件只覆盖**算法引擎容器化**场景。

---

## 1. 拓扑

```
┌────────────────────────── skyengine-net (external) ──────────────────────────┐
│                                                                              │
│   ┌──────────────┐    HTTP/REST     ┌────────────────┐                        │
│   │   engine     │ ───────────────▶ │   mapf (可选)  │  skyengine-mapf-astar │
│   │ sim_server.py│                  └────────────────┘                        │
│   │   :8080      │    HTTP/REST     ┌────────────────┐                        │
│   │              │ ───────────────▶ │   fjsp (可选)  │  skyengine-fjsp-pso    │
│   └──────────────┘                  └────────────────┘                        │
│           ▲                                                                  │
│           │ SSE: /stream/{state,metrics,events}                              │
│           │ HTTP: /sim/{create,play,pause,reset,stop}                        │
└───────────┼──────────────────────────────────────────────────────────────────┘
            │
       上游调用方（Application 的 DockerProxy / 浏览器 / curl）
```

- **engine**：唯一对外暴露端口（`8080`），由 `sim_server.py` 提供 FastAPI 控制接口与 SSE 流
- **mapf / fjsp**：算法微服务镜像。单机测试时由 compose 常驻；平台集成时由 Application 的 `DockerProxy` 按需启停，平时不下发
- 所有服务接入外部网络 `skyengine-net`（必须预先 `docker network create skyengine-net`）

## 2. 镜像构建

`online.dockerfile` 同时支持 CPU / GPU（CUDA 12.9），通过 `BASE_IMAGE` 切换：

```bash
# CPU（默认）
docker build -t skyengine-online:latest -f online.dockerfile .

# GPU（需要 NVIDIA Container Toolkit）
docker build -t skyengine-online:latest \
  --build-arg BASE_IMAGE=nvidia/cuda:12.9.0-cudnn-devel-ubuntu22.04 \
  -f online.dockerfile .
```

要点：
- 基于 `python:3.11-slim-bookworm`，用 `uv` 安装依赖
- 先 `COPY pyproject-online.toml pyproject.toml` + `uv.lock`，再 `uv sync --frozen --no-dev --no-install-project`，利用 Docker 缓存层
- `pyproject-online.toml` 是在线服务精简依赖（不含训练 / 实验脚本所需的重型包），构建产物更小
- 默认 `CMD ["python"]`，实际启动命令由 compose 的 `command` 覆盖为 `uv run python -u sim_server.py`

## 3. 两套 compose 文件

| 文件 | 用途 | engine 启动命令 |
|---|---|---|
| `docker-compose.yaml` | **实验模式**：跑完即退出的批量实验，参数走环境变量（`FJSP_INSTANCE` / `NUM_MACHINES` / `NUM_AGV` …），入口 `run.py` | `run.py` |
| `docker-compose-online.yaml` | **在线模式**：常驻服务，参数走 HTTP JSON，入口 `sim_server.py` | `uv run python -u sim_server.py` |

平台集成（Application 调用）使用 **online** 版本。本文件后续只讲 online。

### 3.1 启动 online 栈

```bash
# 1) 准备外部网络（首次）
docker network create skyengine-net

# 2) 设置环境变量（或写进 .env，用 --env-file 加载）
export SKYENGINE_IMAGE=skyengine-online:latest
export MAPF_IMAGE=skyengine-mapf-astar:latest   # 可选
export FJSP_IMAGE=skyengine-fjsp-pso:latest     # 可选
export CUDA_VISIBLE_DEVICES=0
export SEED=42
export DATA_DIR=/abs/path/to/skyengine/dataset

# 3) 启动（仅 engine）
docker compose -f docker-compose-online.yaml up -d engine

# 4) 或启动全部（engine + mapf + fjsp，单机联调用）
docker compose -f docker-compose-online.yaml up -d
```

停止：

```bash
docker compose -f docker-compose-online.yaml down
```

### 3.2 卷挂载（热更新）

```yaml
volumes:
  - ${DATA_DIR:-./dataset}:/dataset:ro     # 数据集只读
  - ./sim_server.py:/app/sim_server.py:ro   # 改服务代码即生效
  - ./sky_executor:/app/sky_executor:ro     # 改算法代码即生效
  - ./sky_logs:/app/sky_logs                # 日志可写回宿主
  - ./config:/app/config:ro
  - ./test:/app/test:ro
```

`sky_executor` 与 `sim_server.py` 都是 `:ro` bind mount —— 改源码后 `docker compose restart engine` 即可生效，不必重建镜像。只有改动 `pyproject-online.toml` / `uv.lock`（新增依赖）时才需要重新 build。

### 3.3 必需的环境变量

| 变量 | 默认 | 说明 |
|---|---|---|
| `SKYENGINE_IMAGE` | `skyengine-online:latest` | engine 镜像 |
| `MAPF_IMAGE` | `skyengine-mapf-astar:latest` | mapf 镜像 |
| `FJSP_IMAGE` | `skyengine-fjsp-pso:latest` | fjsp 镜像 |
| `DATA_DIR` | `./dataset` | 宿主数据集目录，挂为容器 `/dataset` |
| `CUDA_VISIBLE_DEVICES` | `0` | GPU 选择；空字符串=全部 |
| `SEED` | `42` | 复现性种子 |
| `ENGINE_CMD` | `uv run python -u sim_server.py` | 覆盖 engine 启动命令 |

## 4. sim_server.py 在线 API

`sim_server.py` 是容器内的 FastAPI 服务，端口 `8080`。设计原则：不依赖 `run.py`、参数全部走 HTTP JSON、不读环境变量。

### 4.1 控制端点

| 方法 & 路径 | 作用 | 请求体 |
|---|---|---|
| `POST /sim/create` | 创建仿真环境 | 完整 grid_factory 配置 dict（见 §5） |
| `POST /sim/play` | 启动 / 恢复 | 无 |
| `POST /sim/pause` | 暂停 | 无 |
| `POST /sim/reset` | 重置环境（保留配置） | 无 |
| `POST /sim/stop` | 停止 | 无 |
| `GET  /sim/state` | 当前步与运行标志 | — |
| `GET  /health` | 健康检查 | — |

### 4.2 SSE 流端点

每条流采用「每连接独立订阅、断开自动 unsubscribe」模式，`threading.Queue.get(block=True, timeout=1.0)` 定期释放避免线程泄漏：

| 路径 | 推送内容 |
|---|---|
| `GET /stream/state` | 状态快照（机器 / AGV / 工单） |
| `GET /stream/metrics` | 性能指标（makespan / flowtime / 利用率 …） |
| `GET /stream/events` | 事件流（开始 / 完成 / 异常 …） |

SSE 报文格式：

```
event: <type>
data: <json>

```

curl 示例：

```bash
# 创建（配置体见 §5）
curl -X POST http://localhost:8080/sim/create \
     -H 'Content-Type: application/json' \
     -d @grid_factory.json

curl -X POST http://localhost:8080/sim/play
curl http://localhost:8080/sim/state
curl -N http://localhost:8080/stream/state
```

## 5. create 请求体（grid_factory 配置）

`POST /sim/create` 接收的 JSON 与 `joint_sim/io/use_io.py` 解析逻辑完全一致，`sim_server.create_env_from_config` 是同一份实现的拷贝。关键字段：

```jsonc
{
  "id": "grid_factory",
  "name": "测试产线",
  "topology": {
    "gridWidth": 20,
    "gridHeight": 14,
    "machines": {
      "MACHINE_1_1": { "location": [10, 6], "size": [1, 1], "status": "IDLE" }
    },
    "zones": [
      { "type": "obstacle", "area": { "x": 0, "y": 0, "w": 3, "h": 3 } }
    ]
  },
  "agvs": [
    { "id": 0, "initialLocation": [5, 2], "velocity": 1.0, "status": "IDLE" }
  ],
  "jobs": {
    "job_list": [
      {
        "job_id": 0,
        "operations": [
          {
            "machine_options_with_time": [
              [0, 5], [1, 4], [2, 6]
            ]
          }
        ],
        "arrival_time": 0,
        "due_time": 50,
        "priority": 1
      }
    ]
  }
}
```

要点：
- `topology.zones[type=obstacle]` 会被 `build_obstacle_map` 转成 pogema 字符串障碍图（`#`/`.`），并自动方正化为 `max(gridWidth, gridHeight)` 尺寸；机器格与 AGV 起点格强制保持可通行
- 工序字段**优先** `machine_options_with_time`（标准 FJSP：`List[[machine_id, proc_time]]`），走 `JobConfig(strategy="custom_time")`
- 旧格式 `{machine_id, duration}`（单机器）仍兼容但 **DEPRECATED**，会丢失 per-machine 时间
- `mapf_algorithm == "mapf_gpt"` 时 `GridConfig.observation_type` 切到 `"MAPF"`（dict obs），其它算法保持默认 `POMAPF`（numpy array）

## 6. 与 Application 的联动（DockerProxy）

平台集成时，engine 栈**不常驻**，而是由 Application 的 backend 通过 `DockerProxy` 按需启停：

1. Application 的 `docker-compose.yml` 把宿主 `/var/run/docker.sock` 挂入 backend 容器
2. Application `.env` 中 `SKYENGINE_COMPOSE_PATH` 指向此处的 `docker-compose-online.yaml`
3. backend 收到上游「启动工厂」请求时，`DockerProxy` 调 `docker compose -f $SKYENGINE_COMPOSE_PATH up -d engine`，把 engine 容器拉起在 `skyengine-net`
4. backend 通过 `ENGINE_URL=http://engine:8080` 调用 `/sim/*` 与 `/stream/*`
5. uvicorn 退出时（atexit + `stop_grace_period: 30s`）自动 `compose down` online 栈，不留僵尸容器

> 因此单机调试 engine 时可以直接 `docker compose -f docker-compose-online.yaml up`；上线时不要手动起 engine，交给 DockerProxy 管理即可。

## 7. 常见问题

**Q1：`docker compose up` 报 `network skyengine-net not found`？**
A：online compose 用的是 `external: true` 网络，需先 `docker network create skyengine-net`。

**Q2：engine 启动后 `/health` 通但 `/sim/play` 不前进？**
A：先 `POST /sim/create` 提交配置；`play` 只在已 create 的环境上生效。`GET /sim/state` 看 `step` 是否变化。

**Q3：改了 `sky_executor` 代码没生效？**
A：bind mount 是 `:ro`，`docker compose restart engine` 即可。只有改依赖（`pyproject-online.toml`/`uv.lock`）才需要 `docker compose build engine`。

**Q4：GPU 没用上？**
A：确认镜像用 CUDA base 重建过，且宿主装了 NVIDIA Container Toolkit；`docker run --rm --gpus all skyengine-online:latest nvidia-smi` 验证。

**Q5：online 模式和实验模式（`docker-compose.yaml`）能同时跑吗？**
A：可以，但二者都尝试占用 `8080`。要么改其中一个的端口映射，要么只起其中一个。
