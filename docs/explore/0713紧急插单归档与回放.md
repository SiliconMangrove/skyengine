# 紧急插单归档与回放

本文补充分析日志 schema 1.1 中的插单数据，并说明“实验方案”中的回放行为。

## 1. 数据来源

插单面板不依赖组件本地提交结果。当前运行记录直接来自最新工厂帧的 `insertion_requests`，因此切换页签或组件重新挂载后仍能恢复。

每条记录至少包含：

- `request_id`
- `phase`
- `accepted_step`
- `inserted` 与 `rejected`
- `job_ids`
- `timeline`
- `revision`
- `latency_ms`
- `error`

阶段顺序为：

```text
queued -> replanning -> scheduled -> transporting -> processing -> completed
                      \-> failed
```

## 2. Run schema 1.1

归档顶层新增 `insertionRequests`：

```json
{
  "schema_version": "1.1",
  "frames": [],
  "metricsTimeline": [],
  "events": [],
  "insertionRequests": [],
  "summary": {
    "insertion_count": 0,
    "insertion_completed": 0,
    "insertion_failed": 0
  }
}
```

保存归档时，前后端都会按 `request_id` 从所有帧中汇总记录，并保留阶段更新最完整的版本。显式传入的 `insertionRequests` 会与帧内记录合并。运行数据库只保存归档索引和内容，不属于源码提交。

## 3. 实验方案回放

实验方案面板加载归档时：

1. 恢复 `frames`、`metricsTimeline` 和 `events`。
2. 若旧帧没有 `insertion_requests`，将顶层 `insertionRequests` 注入最后一帧，供插单面板恢复记录。
3. 回放页显示该 Run 的插单数量、完成数和失败数。
4. “插单跳转”从事件流提取插单相关事件，点击后跳到对应 step，便于查看到达、重规划、抢占、恢复和完成过程。

## 4. 兼容性

- schema 1.0 或外部导入文件没有 `insertionRequests` 时按空数组处理。
- 后端会重新计算 summary，不信任导入文件自带的插单统计。
- StaticFactory 不支持真实插单，不再返回模拟成功结果。
- 只有运行中的 `PSO + nearest` 在线工厂允许提交。

## 5. 验证

```powershell
pytest application/backend/analysis/test_service.py
cd application/frontend
npm run build
```

在线验收时应归档一次包含已完成或失败插单的运行，重新加载该 Run，并确认插单记录和事件跳转仍然存在。
