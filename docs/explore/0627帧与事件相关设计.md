## 0. 背景与核心结论

设计日志 / 分析体系时，必须先回答一个上游问题：**engine 到底吐什么、什么时候吐？** 本文先把现状摸清，再给出契约方向。

**核心结论**：

1. 容器化 SkyEngine 采用 **frame 每步必产 + 事件由帧间 diff 派生** 的范式。engine 自身不"写日志"，事件是状态变迁的副产品。
2. 事件产出有明确边界：**只有"语义跃迁"才算事件**，连续量（位置、进度、队列长度）靠每步的 frame/metrics 流承载。
3. 当前 diff 只覆盖 2 个维度（机器 op 跃迁、transfer 起止），给出 4 种事件类型，架构上易于扩展。
4. StaticFactory 走完全不同的范式：**硬编码剧本，事件按 step 精确匹配，不走 diff**。两种范式都必须被契约容纳。

## 一、容器化 SkyEngine 的事件流现状

### 1.1 整体串联

主循环 `_run_loop`（`sim_server.py:583`）跑在一个 daemon 线程里，每次迭代（约 0.5s/步）：

```
coordinator.decide()          # 算法容器（fjsp/mapf）返回动作
   ↓
env.step(actions)             # GridFactoryEnv 推进一步
   ↓
self.step += 1
   ↓
frame = self._build_frame()   # 构造权威状态帧             sim_server.py:614
   ↓
self._diff_and_emit(frame)    # ★ 帧间 diff 派生事件        sim_server.py:622
   ↓
broadcast state + metrics     # 每步必发的两条流
```

**关键：frame 每步都建、每步都广播；事件不是每步都发，只在 diff 检测到"值得说的变迁"时才发。**

### 1.2 三条 SSE 流的分工

engine 容器内 `sim_server.py` 暴露三条 SSE 端点，`DockerProxy` 用 `httpx` **反向 SSE 订阅**逐行透传（不是轮询、不是 websocket）：

| 流 | 端点 | 频率 | 内容 |
|---|---|---|---|
| state | `/stream/state` | 每步一帧 | 完整 frame（AGV/机器/job/transfer） |
| metrics | `/stream/metrics` | 每步一条 | `{status, step, metrics:{...}, metrics_reward}` |
| events | `/stream/events` | **仅跃迁时** | `{timestamp, type, message, level}` |

晚连接者订阅时，state 流补发最新帧、events 流补发一条 `sim_started`（`sim_server.py:254-259`）。

## 二、Frame：engine 的权威状态快照

`_build_frame`（`sim_server.py:544-563`）从 env 状态归一化出一个 dict：

```jsonc
{
  "timestamp": "T+{step}s",
  "env_timeline": "<pogema 时间线字符串>",
  "grid_state": {
    "positions_xy": [[x, y], ...],     // AGV 坐标列表
    "is_active":     [true, false, ...] // AGV 是否在执行任务
  },
  "machines": [
    { "id": 1, "current_op": {"job_id":..,"op_id":..}, "queue_length": .., ... }
  ],
  "jobs": [ ... ],
  "active_transfers": [
    { "task_id": .., "job_id": .., "op_id": .., "source": [..], "destination": [..] | null }
  ]
}
```

注意：**machines 是 list，machine.id 是 int**（与 StaticFactory 的 `dict{"M1":...}` 不同，见第五节）。

frame 是后续 diff 的"比对对象"，也是前端动画与 Hook 图表的唯一数据源。

## 三、事件产出：diff-derived 的核心机制

### 3.1 判定函数 `_diff_and_emit`（`sim_server.py:302-371`）

**这是"是否产出事件"的唯一裁判。** 它接收当前 frame，与上一步的 `_prev_frame` 比对，**只盯两个字段**的变迁，其余变化一律不产事件。

#### 首帧特殊处理（`:315-317`）
`_prev_frame is None` 时（episode 第一步）**不发任何事件**，只把当前 frame 存为基线，后续才开始 diff。

#### 规则 1：机器 `current_op` 状态机（`:324-349`）
先用 `_op_key(op)`（`:319-322`）把 current_op 归约为 `(job_id, op_id)`，**忽略 step_done 等动态字段**，避免渐进字段误触发。然后比对每台机器 old vs cur：

| old（上一步） | cur（本步） | 产出事件 | message | level |
|---|---|---|---|---|
| `None` | `(j,o)` | `machine_start_op` | "M{id} 开始 Job{j}-Op{o}" | info |
| `(j,o)` | `None` | `machine_idle` | "M{id} 完成加工，进入空闲" | success |
| `(j,o)` | `(j',o')` 且不同 | `machine_start_op` | "M{id} 切换至 Job{j'}-Op{o'}" | info |
| 不变 | 不变 | 无事件 | — | — |

#### 规则 2：`active_transfers` 的 task_id 集合差（`:351-369`）
把 prev / cur 的 task_id 各做成集合，做对称差：

| 集合关系 | 产出事件 | message | level |
|---|---|---|---|
| `tid ∈ cur 且 tid ∉ prev`（新出现） | `transfer_started` | "AGV 取货 Task{tid} (Job..-Op..) {src} → {dst}" | info |
| `tid ∈ prev 且 tid ∉ cur`（消失） | `transfer_completed` | "AGV 完成 Task{tid}" | success |

比对完，`self._prev_frame = frame`（`:371`）滚动基线。

### 3.2 事件封装 `_emit_event`（`:290-300`）

```python
event = ("event", {
    "timestamp": f"T+{self.step}s",
    "type": etype,        # machine_start_op / machine_idle / transfer_started / transfer_completed
    "message": message,    # 人读的中文描述
    "level": "info",       # info / success（当前只用这两个级别）
})
self._broadcast(self._events_subs, event)
```

### 3.3 当前事件词汇表

| type | 触发条件 | level |
|---|---|---|
| `sim_started` | 订阅时补发（非 diff 产生） | info |
| `machine_start_op` | 机器 op 从无到有 / 切换 | info |
| `machine_idle` | 机器 op 从有到无 | success |
| `transfer_started` | 新 task_id 出现 | info |
| `transfer_completed` | task_id 消失 | success |

## 四、设计哲学：事件 vs 连续流

**事件产出判定的核心原则：只有"语义跃迁"才算事件，连续量靠流承载。**

| frame 字段 | 变化时是否产事件 | 承载方式 |
|---|---|---|
| `machines[].current_op` | ✅ 产 | events 流（离散） |
| `active_transfers[].task_id` | ✅ 产 | events 流（离散） |
| `grid_state.positions_xy`（AGV 坐标） | ❌ 不产 | state 流（每步） |
| `machines[].queue_length` | ❌ 不产 | state 流 + metrics 流 |
| `jobs[].progress` | ❌ 不产 | state 流 + metrics 流 |

**本质**：events 流是 state 流的**有损压缩**——丢掉连续量，只留跃迁。这样：
- 人看事件面板不会被噪声淹没（每条都是"发生了什么"）
- 分析侧要还原连续量时，回读 state/metrics 流即可
- 离散事件适合做"异常检测 / 关键节点标注"，连续流适合做"趋势图"

**判定方式**：纯状态机/集合 diff，**没有阈值、没有计数、没有时序条件**——某一步检测到跃迁就立刻发。

## 五、StaticFactory 的对照：硬编码剧本范式

StaticFactory（`application/backend/core/StaticFactoryProxy.py`）走完全不同的路：

### 5.1 数据来源
- AGV 轨迹硬编码在 `AGV_TRAJECTORIES`（`:29-85`）
- Job 剧本硬编码在 `JOB_SCRIPT`（`:100-134`）
- 事件硬编码在 `EVENTS_LOG`（`:138-145`）

### 5.2 事件 schema（与容器化 engine 完全不同）

```jsonc
{ "step": 5, "title": "AGV-1 到达 M3", "message": "...", "type": "info" }
```

| 维度 | 容器化 engine | StaticFactory |
|---|---|---|
| 主键 | `timestamp`（"T+{step}s"） | `step`（int） |
| 描述字段 | `message` | `title` + `message` |
| 分类字段 | `type`（业务语义枚举） + `level` | `type`（日志级别：info/success/task/error） |
| 触发方式 | 帧间 diff 自动派生 | 按 step 精确匹配（`get_events(step)`，`:282-284`） |
| 传输通道 | 独立 `/stream/events` | 嵌在 state snapshot 的 `events` 字段里 |

### 5.3 本质差异
- 容器化 engine：**事件是状态变迁的副产品**，diff 自动产出，作者写剧本时不预知会有哪些事件
- StaticFactory：**事件是剧本作者预先编排的台词**，固定在某个 step 上发生，与状态变迁无因果关系

**两种范式都必须被统一契约容纳**——这是契约设计的难点。

## 六、契约方向（待 0627 日志方案定稿后正式落地）

基于现状，统一契约的取舍方向：

### 6.1 事件信封（envelope）统一
无论哪种 engine，事件最终都归一成同一字段 shape。建议字段：

```jsonc
{
  "step": 42,                       // 强制：步号（容器化从 timestamp 反推，Static 原生）
  "timestamp": "T+42s",             // 强制：引擎时间线
  "category": "machine_op",         // 强制：统一分类（见 6.2）
  "type": "machine_start_op",       // 引擎相关：原始事件类型
  "level": "info",                  // 强制：info / success / warning / error
  "title": "机器开始加工",           // 强制：短标题（≤20 字）
  "message": "M3 开始 Job2-Op1",    // 可选：详细描述
  "payload": { ... }                // 引擎相关：原始业务字段
}
```

**取舍**：`category` 统一、`type` 引擎相关。前端列表按 `category` 聚合，详情看 `type`/`payload`。

### 6.2 统一分类（category）词汇表
把引擎相关的事件类型映射到固定分类：

| category | 含义 | 容器化 engine 的 type | StaticFactory 的 type |
|---|---|---|---|
| `lifecycle` | 回合生命周期 | `sim_started` | （可补 `started`/`finished`） |
| `machine_op` | 机器加工状态变迁 | `machine_start_op`, `machine_idle` | `task` |
| `transfer` | 搬运任务起止 | `transfer_started`, `transfer_completed` | `task`（语义靠 title 区分） |
| `system` | 系统级提示 | — | `info` / `success` |
| `error` | 异常 | （未来 `machine_fail` 等） | `error` |

### 6.3 事件产出职责归属
- **真实 engine（容器化）**：事件在 engine 侧由 diff 派生，proxy/前端只透传，不自行造事件
- **StaticFactory**：事件由剧本作者预编排，proxy 按步吐出
- **前端**：**永远不自行产出"业务事件"**，只负责展示与归档。前端可产出的只有 UI 级事件（如"用户点了执行"），归 `system` 类且与 engine 事件分流

### 6.4 扩展点
现有容器化 engine 的 diff 只覆盖 2 个维度。**未来加事件类型 = 给 `_diff_and_emit` 加规则分支**，架构已铺好，候选扩展：
- `job_completed`：job 的所有 op 完成（watch `jobs[].is_completed`）
- `machine_fail` / `agv_block`：异常态跃迁（需 engine 先暴露这些状态）
- `deadline_risk`：基于时序/阈值的预警（**注意：这是首个非纯跃迁的判定，需评估是否引入"规则引擎"层**）

## 七、与日志存储的衔接

本文定义的 frame + event 是 `0627日志更新.md` 里 RunData 的数据来源：

```
engine 每步产 frame      →  RunData.frames[]
engine 每步产 metrics    →  RunData.metrics_timeline[]
engine 跃迁产 event      →  RunData.events[]
```

**归档时机**：episode 结束（`terminations.get("job_done")` 或步数上限，`sim_server.py:644`）时，把累积的三条流打包成一份完整 Run，走 history 模块落盘。

**当前断点**（待补）：
- `sim_server.py` 没有把 frame/metrics/events 累积成 episode 级 buffer，只在内存里"最新帧 + 实时广播"
- `DockerProxy` 只透传，不落盘
- `history` 模块的 `append_run_data` 从未被调用，RunData.events 永远空

**补法**：要么在 engine 侧 episode 结束时一次性 POST 完整 Run 给后端，要么在 DockerProxy 订阅期间累积、episode 结束时落盘。后者更解耦（engine 不感知归档），推荐。
