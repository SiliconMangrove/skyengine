# 0622 更新设计 — StaticFactory 静态剧本驱动三模块落地

> 范围：**0619 设计的三个模块（① Machine & Job 富 frame / ② 在线自定义绘图 / ③ 离线日志分析）全部先在 `StaticFactoryProxy` 上以"静态剧本"方式落地一份**，作为 sim_server / DockerProxy 改造的对照参考实现与无 Docker 开发床。
> 策略：StaticFactory 先跑通 → 验证前端三件套（Panel / Chart 模板 / 离线看板） → 复刻到 DockerProxy（sim_server）与 PacketFactory。

## 0. 三件事与 StaticFactory 的关系

| 序号 | 工作（0619 原始定义） | StaticFactory 落地形态 | 后续复刻目标 |
| --- | --- | --- | --- |
| ① | Machine & Job 状态细化 | `JOB_SCRIPT` 常量 + `FactorySimulator` 派生富 frame | sim_server `_build_frame` |
| ② | 在线自定义绘图（Hook + 模板 + 配置三层） | 静态 metrics 喂 Hook，跑通模板与配置驱动看板 | DockerFactory MetricsPanel 重写 |
| ③ | 离线日志分析（华佗迁移） | StaticFactory 产 JSONL 静态日志 → 走完整离线管线 | sim_server 日志落盘 + 批处理下游 |

**为什么先做 StaticFactory**：
1. 零依赖（不开 Docker、不连引擎），前端三模块可独立验证
2. 确定性剧本 = 可复跑的回归基线，sim_server 改完逐字段对照
3. 三模块的数据契约在这里一次性定稿，后续 port 只换数据源、不改 schema

**优先级**：① → ② → ③。①是 frame 地基，②是渲染地基，③依赖②的模板复用。

## 1. 模块①：Machine & Job 富 frame（静态剧本版）

### 1.1 现状

- `StaticFactoryProxy.py` 现有静态注入三块：
  - `AGV_TRAJECTORIES`（55 步 AGV 坐标常量）
  - `EVENTS_LOG`（6 条事件常量）
  - `FactorySimulator.generate_machine_states(step)` 分阶段返回 `{Mx:{id,status,load}}`
- machine 缺：`current_op` 对象（只有老的二元组语义、其实根本没有）、`queue_length`、显式 `status`
- **顶层 `jobs` 完全没有**
- 前端消费端（`MachineStatusPanel` / `JobStatusPanel` / store 选中联动）已写好，等数据源

### 1.2 目标 frame schema（与 0619 §1.2 一字对齐）

```json
{
  "timestamp": "T+12s",
  "env_timeline": "12",
  "grid_state": { "positions_xy": [[8,11]], "is_active": [true] },
  "machines": {
    "M1": {
      "id": "M1", "status": "WORKING", "load": 60,
      "current_op": {
        "job_id": 1, "op_id": 1,
        "index_in_job": 1, "total_in_job": 3,
        "step_done": 3, "proc_time": 6
      },
      "queue_length": 1
    },
    "M2": { "id": "M2", "status": "IDLE", "load": 0, "current_op": null, "queue_length": 0 }
  },
  "jobs": [
    {
      "job_id": 1, "release": 0, "due": 40,
      "is_completed": false, "completion_time": -1,
      "progress": { "done": 1, "total": 3 },
      "ops": [
        { "op_id": 0, "status": "FINISHED",   "proc_time": 5, "assigned_machine": 0,
          "arrive_machine_at": 0, "start_process_at": 0,  "finish_process_at": 5,  "wait_for_machine_time": 0 },
        { "op_id": 1, "status": "PROCESSING", "proc_time": 6, "assigned_machine": 1,
          "arrive_machine_at": 5, "start_process_at": 5,  "finish_process_at": -1, "wait_for_machine_time": 0 },
        { "op_id": 2, "status": "PENDING",    "proc_time": 4, "assigned_machine": null }
      ]
    }
  ],
  "active_transfers": [],
  "events": []
}
```

字段口径以 0619 §1.2 为权威，StaticFactory 不自创字段。

### 1.3 改造点（仅 `StaticFactoryProxy.py` 一个文件）

**a) 新增模块级常量 `JOB_SCRIPT`**

仿 `AGV_TRAJECTORIES` / `EVENTS_LOG` 的风格，定义一条静态 Job-Op 时间轴：

- 3 条 Job（J1/J2/J3）× 每条 3 道 op，覆盖 0–54 步
- `due` 故意设置：J1 超期 / J2 准时 / J3 无交期，验证 `JobStatusPanel` 高亮分支
- 时间戳用 step 单位（与 `env_timeline` 对齐），不引入真实秒
- **确定性**：不引入 random，保证可复跑、可对照
- 剧本与 AGV 折返点 / machine 阶段对齐（折返点 = Op 切换点 = AGV 装载事件）

**b) `FactorySimulator` 新增方法**

```python
@staticmethod
def generate_jobs(step: int) -> list:
    """从 JOB_SCRIPT 派生当前 step 的 jobs 快照。
    - 遍历每个 op，按 step 判定 status (PENDING/PROCESSING/FINISHED)
    - 计算 progress.done/total、is_completed、completion_time
    """

@staticmethod
def generate_machine_states(step: int) -> dict:
    """扩字段：现有 {id,status,load} 基础上加 current_op 对象 + queue_length。
    current_op 由 generate_jobs(step) 反查：status==PROCESSING 且 assigned_machine==本机 的 op。
    queue_length 派生：arrive_machine_at <= step && status==PENDING && assigned_machine==本机 的 op 数。
    """
```

派生关系（**单向数据流**，避免循环）：
```
JOB_SCRIPT (常量)
   ↓ generate_jobs(step)
jobs[]
   ↓ 反查 PROCESSING / PENDING
machines[*].current_op / queue_length
```

`load` / `status` 保留现有阶段化逻辑，老前端降级显示不崩。

**c) `_build_state_snapshot_direct` 拼装**

在现有快照上追加 `jobs` 和 `env_timeline`：
```python
snapshot = {
    "timestamp": f"T+{self._current_step}s",
    "env_timeline": str(self._current_step),   # 新增，对齐 sim_server
    "grid_state": {...},
    "machines": machine_states,                 # 已含 current_op / queue_length
    "jobs": FactorySimulator.generate_jobs(self._current_step),  # 新增
    "active_transfers": [],
    "events": events,
}
```

**d) 不改的部分**：`_run_mock` 主循环、`get_metrics_snapshot`、`get_control_status`、队列缓冲、SSE 接口。

### 1.4 前端零改动验证清单

| 能力 | 验证路径 |
| --- | --- |
| `MachineStatusPanel` 双进度条（op 步数 + JOB 进度） | StaticFactory → 启动 → machines tab |
| `MachineStatusPanel` 空闲降级（`current_op == null`） | 剧本里 J2 完成后到 J3 开始前 |
| `JobStatusPanel` op 时间轴三态着色 | jobs tab，整段运行 |
| `JobStatusPanel` 交期超期高亮 | J1 故意 `due < completion_time` |
| `JobStatusPanel` 展开明细（arrive/start/finish/wait） | 点击单条 Job |
| machine ↔ Job 双向联动选中 | 点 machine 的 current_op ↔ 点 Job 的 PROCESSING op |
| `normalizeSnapshot` list/dict 双形态 | StaticFactory 走 dict 分支，DockerFactory 走 list 分支 |

## 2. 模块②：在线自定义绘图（静态 metrics 喂 Hook）

### 2.1 现状与问题

- StaticFactory 已经在产 metrics（`FactorySimulator.generate_metrics_data(step)` → SSE `/stream/metrics`）
- `monitor.js` 已累积 `metricsTimeline`（step × metrics）
- `MetricsPanel.vue` 现状是反例：13 个指标写死在 `SIM_METRICS_META`、折线写死两条线、热力图用 div grid 不用 ECharts
- 新增指标 / 换图表样式都要改组件代码 → 违反师兄三条铁律

### 2.2 三层落地（StaticFactory 作为样板间）

**数据层 Hook**（`frontend/src/hooks/metrics/*.js`）
```js
// hooks/metrics/machine_util.js
export default {
  id: 'machine_util', label: '机器利用率', unit: '%',
  series: (monitor) => monitor.metricsTimeline.map(p => ({
    x: p.step,
    y: (p.metrics?.machine_utilization ?? 0) * 100,
  })),
}
```
- 第一批 Hook 直接搬 `SIM_METRICS_META` 那 13 个指标 + episode 终态（makespan、occupancy）
- StaticFactory 的 metrics 字段是确定的（`generate_metrics_data` 写死），所以 Hook 在这里跑出来的曲线是**可复现基线**，sim_server 改完后逐 Hook 对照曲线形状即可
- 新增指标只加 Hook 文件 + 注册，不动模板

**模板层**（`frontend/src/components/charts/*.vue`）
- `LineChart` / `BarChart` / `StackedAreaChart` / `HeatmapChart`
- 每个组件**只接收标准 series** `[{name, data:[{x,y}], color?}]`，不知道指标名
- 把现有 `createMiniOption/createBigOption`（`MetricsPanel.vue:360-395`）的 ECharts option 抽进来
- occupancy 的 div grid 换 ECharts heatmap

**配置层**（`config/dashboards/static_factory.json`）
```json
{
  "id": "static_factory_default",
  "title": "StaticFactory 看板（样例）",
  "cards": [
    {"hook":"machine_util","template":"line","yMax":100},
    {"hook":"agv_loaded","template":"line","overlay":["agv_busy"]},
    {"hook":"swap_conflict","template":"bar"},
    {"hook":"occupancy","template":"heatmap"}
  ]
}
```
- StaticFactory 的看板配置是**最小可用集**，用于验证配置驱动的渲染链路
- DockerFactory 复刻时只新建 `docker_factory.json`，Hook / 模板完全复用

### 2.3 StaticFactory 改造路径

1. 新建 `hooks/metrics/` 目录，从 `SIM_METRICS_META` 迁出 13 个指标成 Hook 文件
2. 新建 `components/charts/`，把 `MetricsPanel.vue` 的 ECharts option 抽过去
3. `MetricsPanel.vue` 改成读 `config/dashboards/static_factory.json` 渲染卡片（StaticFactory 分支重写，PacketFactory 旧分支保留）
4. `StaticFactoryManage.vue` 接入新 MetricsPanel（替换或新增 tab）

### 2.4 为什么 StaticFactory 先做

- metrics 字段稳定（写死），Hook 输出可预测 → 验证 Hook 契约的最佳环境
- 不依赖 episode 终态的随机性 → 模板层的 axis / legend / tooltip 调参有可复现基线
- 验完后 DockerFactory 只需新增 Hook（新字段）+ 新增 dashboard JSON，模板层零改动

## 3. 模块③：离线日志分析（静态 JSONL 喂完整管线）

### 3.1 关键洞察

StaticFactory 跑一遍 55 步 = 一次"伪 episode"。把每步的 frame + 派生事件落成 JSONL，就得到一份**确定性离线日志样本**，可端到端验证华佗管线（上传 → 解析 → 离线 Hook → 复用模板层渲染）。

→ **StaticFactory 改造最小切口**：`_run_mock` 每推一帧时同步写一行 JSONL，跑完产出 `sky_logs/episode_static_{ts}.jsonl`。

### 3.2 数据流（StaticFactory 样板版）

```
StaticFactory._run_mock (55 步)
   ↓ 每步同时写 JSONL 一行
sky_logs/episode_static_{ts}.jsonl
   ↓ POST /api/logs/upload (前端手动触发或自动)
run_id = xxx
   ↓ POST /api/logs/{run_id}/analyze
离线 Hook 计算 metrics_dict
   ↓ 复用模块②同一套 Chart 模板
views/analysis/ (离线看板)
```

### 3.3 JSONL 行 schema（与 0619 §3.4 对齐）

```json
{"step":12,"type":"frame","frame":{...完整 frame，同模块① schema...}}
{"step":12,"type":"event","event":{"title":"...","message":"...","type":"success"}}
{"step":12,"type":"metrics","metrics":{...同 generate_metrics_data 输出...}}
```

- 一行一个 JSON 对象，`type` 区分 frame / event / metrics
- 最终步追加一行 `{"type":"episode_summary", ...}` 携带 makespan / utilization 等终态
- schema 与 sim_server 未来落盘格式**一字对齐**（0619 §3.4 / §6.5 待评审项统一定稿）

### 3.4 离线 Hook（契约对齐在线）

```js
// hooks/offline/machine_util.js — 与在线同名 Hook 接口一致
export default {
  id: 'machine_util',
  series: (parsedLog) => parsedLog.frames.map(p => ({
    x: p.step,
    y: (p.metrics?.machine_utilization ?? 0) * 100,
  })),
}
```

- 前端配置层完全复用（看板 JSON 只写 `hook:"machine_util"`，由数据源类型决定调在线还是离线版本）
- StaticFactory 的离线日志因为确定性，可以生成"黄金基线 metrics_dict"做回归对照

### 3.5 后端新增（`application/backend/logs/`）

首期最小集（StaticFactory 验证用）：
- `POST /api/logs/upload` — 接收 `.jsonl`，校验 schema，落 `analysis_runs/`
- `POST /api/logs/{run_id}/analyze` — 解析日志，跑离线 Hook 计算指标字典
- `POST /api/logs/export` — 按 run_id 打包下载（可选）
- 注册到 `core/RouteRegistry.py`

### 3.6 前端新增

- `views/analysis/` 离线看板页，复用 `components/charts/*` 模板
- 上传组件 + run 列表 + 时间拖拽（ECharts `dataZoom`）

### 3.7 为什么 StaticFactory 先做

- 离线管线（上传 / 解析 / Hook / 渲染）一旦跑通，sim_server 只需"产 JSONL"这一个动作就能接入
- 确定性日志 → 可作为 CI 回归样本，防止后续改动破坏解析契约
- StaticFactory 跑一次产一份日志，无需启动 Docker，开发环路最短

## 4. 三模块在 StaticFactory 上的协同

```
              ┌────────────────────────────────────┐
              │  JOB_SCRIPT (常量) + FactorySimulator │
              └──────────────┬─────────────────────┘
                             │ generate_jobs / generate_machine_states
                             ↓
              ┌────────────────────────────────────┐
              │  富 frame (模块①)                   │
              │  - SSE 实时推 → 前端 Panel          │
              │  - 同步落 JSONL (模块③输入)         │
              │  - metrics 喂 Hook (模块②输入)      │
              └──────┬───────────────┬─────────────┘
                     │               │
                     ↓               ↓
        ┌────────────────────┐  ┌──────────────────────┐
        │ 模块② 在线看板     │  │ 模块③ 离线看板       │
        │ Hook + 模板 + 配置 │  │ 上传 → 解析 → 同模板 │
        └────────────────────┘  └──────────────────────┘
```

三个模块共用 frame schema（模块①定）、共用 Chart 模板（模块②定）、共用 JSONL schema（模块③定）。StaticFactory 把三份契约一次性钉死，sim_server 复刻时只是数据源替换。

## 6. 设计纪律（继承自师兄三条铁律 + 0619 补充）

1. **静态剧本是数据，不是逻辑**：`JOB_SCRIPT` 只描述"什么时候发生什么"，状态派生由 `FactorySimulator` 纯函数完成，不引入副作用
2. **字段口径单一权威**：以 0619 §1.2 为准，StaticFactory 不自创字段名；有歧义改 0619，不改 0622
3. **前端零分支**：StaticFactory / DockerFactory / PacketFactory 共用同一套 Panel / Chart 模板 / 离线看板，差异只在 `normalizeSnapshot` 归一化层与 Hook 数据源
4. **不写兼容 shim**：老 `current_op` 二元组一次性切对象格式，不做"如果是数组就兼容"的退化逻辑（`normalizeSnapshot` 只处理 list/dict 容器形态，不处理字段结构差异）
5. **指标渲染纪律（模块②铁律）**：任何指标渲染需求先写 Hook、再选模板、最后改配置，不允许直接在组件里写死指标名 + ECharts option
6. **在线离线共用渲染内核**：在线 Hook 与离线 Hook 同 id 同契约，仅数据源不同
