## 0. 现状回顾

当前 `_diff_and_emit` 只盯 **2 个 frame 字段**的变迁，产出 **4 种业务事件 + 1 种订阅补发事件**：

| 已有 type | 来源 | 触发判定 |
|---|---|---|
| `sim_started` | 订阅补发（非 diff） | 新订阅者连上 events 流 |
| `machine_start_op` | diff：machine.current_op | `None → (j,o)` 或 `(j,o) → (j',o')` |
| `machine_idle` | diff：machine.current_op | `(j,o) → None` |
| `transfer_started` | diff：active_transfers.task_id 集合 | 新 task_id 出现 |
| `transfer_completed` | diff：active_transfers.task_id 集合 | task_id 消失 |

**设计哲学**（不变）：只有"语义跃迁"才算事件；连续量（坐标、进度、队列长度）由 state/metrics 流承载。

**扩展目标**：在不违背这一哲学的前提下，把更多**离散跃迁**纳入 diff，并谨慎评估是否引入**阈值类**事件。

## 一、可派生事件的 frame 字段盘点

把 `_build_frame`（`sim_server.py:464-563`）产出的字段过一遍，标出"是否离散跃迁 / 是否已被用 / 是否需补字段"：

| frame 区段 | 字段 | 类型 | 现状 | 潜在事件 |
|---|---|---|---|---|
| `grid_state` | `is_active[]` | bool 数组 | 已用于派生 `agv_list` | AGV 进入/退出任务态 |
| `grid_state` | `positions_xy[]` | 连续 | 未用（设计上**不应**产事件） | — |
| `machines[]` | `current_op` | 离散跃迁 | ✅ 已用 | （已覆盖） |
| `machines[]` | `status` | 派生于 current_op | 未直接 diff（等价） | 冗余 |
| `machines[]` | `queue_length` | 整数 | 未用 | 队列长度阈值（阈值类） |
| `jobs[]` | `is_completed` | bool | **未用** | `job_completed` |
| `jobs[]` | `ops[].status` | `"PENDING"\|"PROCESSING"\|"FINISHED"` | **未用** | `op_status_changed`（部分与 machine 事件重叠） |
| `jobs[]` | `release` | 数值 | 未用 | `job_released`（任务投放） |
| `jobs[]` | `due` | 数值（deadline） | 未用 | `deadline_risk` / `deadline_missed`（阈值/时序类） |
| `jobs[].ops[]` | `arrive_machine_at` / `wait_for_machine_time` | 数值 | 未用 | `op_long_wait`（阈值类） |
| `active_transfers[]` | `task_id` 集合 | 离散跃迁 | ✅ 已用 | （已覆盖） |
| `active_transfers[]` | `destination` | `null` ↔ 解析值 | 未用 | `transfer_destination_resolved`（"动态任务目标确定"） |
| `env_timeline` | 数值 | 连续 | 未用 | episode 生命周期（与 step 配合） |

**frame 外、引擎内可达**（需在 `_build_frame` 补字段或直接读 `pogema` 才能派生）：

| 引擎对象 | 字段 | 用途 |
|---|---|---|
| `agv.current_task`（`AGV` 类，`structure.py:161`） | RoutingTask \| None | AGV 进入/退出任务态、`agv_assigned`/`agv_freed` |
| `RoutingTask.create_time / assign_time / finish_time`（`structure.py:123-158`） | 数值 | transfer 等待/超时（阈值类） |
| `RoutingTask.dynamic`（property） | bool | 动态任务是否已解析目标 |
| `infos["metrics"]`（每步） | 标量 | 见下节"指标驱动" |

## 二、候选事件清单（按判定方式分两类）

### 2.1 类别甲：纯 diff（状态机/集合变迁）—— 与现有哲学一致

这类**零阈值、零时序条件**，某步检测到跃迁就发，与现有 4 类完全同构，**应优先落地**。

| 候选 type | 触发判定 | level | message 模板 | 备注 |
|---|---|---|---|---|
| `job_completed` | `jobs[].is_completed` 由 false → true | success | "Job{j} 完成（{n}/{n} 工序）" | 高价值，episode 收口 |
| `job_released` | `env_timeline` 跨过 `job.release`，且该 job 此前未触发 | info | "Job{j} 投放（release={r}）" | 可选；release=0 的 job 在第 0 步触发 |
| `agv_assigned` | 某 AGV 的 current_task 由 None → 有值 | info | "AGV{i} 接单 Task{tid}" | 需 frame 暴露 AGV task_id |
| `agv_freed` | 某 AGV 的 current_task 由有值 → None | success | "AGV{i} 完成任务进入空闲" | 同上 |
| `transfer_destination_resolved` | active_transfers 某 t 的 destination 由 null → 有值 | info | "Task{tid} 目标确定为 {dst}" | 体现"动态调度"语义 |
| `episode_completed` | `terminations.job_done` 由 false → true | success | "Episode 完成：全部 Job 完成" | 终止语义，与 step 上限区分 |
| `episode_truncated` | `step >= max_steps` 且 `job_done` 仍为 false | warning | "Episode 截断：步数上限" | 当前 `_run_loop` 只判 step，应转为事件 |

**实现成本**：
- `job_completed / job_released`：frame 已有字段，**零新增字段**。
- `episode_completed / episode_truncated`：`_run_loop`（`sim_server.py:644`）已有判定，只缺"作为事件发出"，~5 行。
- `agv_assigned / agv_freed`：需在 `_build_frame` 的 `grid_state` 加 `"tasks": [task_id|null, ...]`（或新增 `agvs[]` 区段），然后 diff。改动 ~10 行。
- `transfer_destination_resolved`：frame 已有 `destination`，零新增字段。

### 2.2 类别乙：阈值/规则类 —— 哲学分叉点

这类**不是纯跃迁**，依赖阈值、时序、连续量比较。**当前 `_diff_and_emit` 无此能力**，引入它意味着新增一层"规则引擎"。需用户拍板是否引入。

| 候选 type | 判定 | level | 备注 |
|---|---|---|---|
| `deadline_risk` | 某 job 未完成且 `env_timeline > due * α`（α≈0.8） | warning | 需去抖：同一 job 只发一次 |
| `deadline_missed` | 某 job 在 `env_timeline > due` 时仍未完成 | error | 一次性 |
| `op_long_wait` | 某 op `wait_for_machine_time > τ` 且未开始加工 | warning | τ 建议相对 makespan 归一 |
| `machine_queue_backlog` | 某 machine `queue_length` 跨阈值 K | warning | 跨阈值+回落后再跨才重发 |
| `agv_swap_conflict` | `metrics.swap_conflict_count` 当步增量 > 0 | warning | 指标驱动，每步可多次 |
| `agv_stationary_loaded` | `metrics.tasked_stationary_count` > 0 持续 N 步 | warning | 时序累积 |
| `transport_blocking` | `metrics.transport_blocking_delay_mean` 跨阈值 | warning | 指标驱动 |
| `machine_starved` | `metrics.machine_waiting_for_inbound_transfer_ratio > β` | warning | 指标驱动 |

**两种实现路径**：
- **(P1) 内嵌规则**：在 `_diff_and_emit` 末尾追加一段"指标/阈值检查"，所有规则硬编码在 sim_server。优点：单文件；缺点：函数膨胀、规则与状态机混杂。
- **(P2) 规则引擎层**：新建 `_rule_check(frame, metrics, ctx)`，与 `_diff_and_emit` 并列，由 `_run_loop` 顺序调用。`ctx` 持有"本 episode 已发的阈值事件集合"用于去抖。优点：职责分离、规则可插拔；缺点：多一层抽象。

**建议**：阈值类**全部推迟**到规则引擎层（P2），不要污染 `_diff_and_emit`。但**是否引入规则引擎层本身**需用户决策——它是 `0627帧与事件相关设计.md` §6.4 标记的"首个非纯跃迁判定"。

## 三、推荐落地优先级

| 阶段 | 事件 | 类别 | 理由 |
|---|---|---|---|
| **P1（先做，零风险）** | `job_completed`、`episode_completed`、`episode_truncated` | 甲 | frame 已有字段，纯 diff，episode 收口语义最刚需 |
| **P1** | `transfer_destination_resolved` | 甲 | frame 已有 destination，零新增字段，体现动态调度 |
| **P2（小改 frame）** | `agv_assigned`、`agv_freed` | 甲 | 需 frame 暴露 AGV task_id，补完 AGV 维度生命周期 |
| **P2（可选）** | `job_released` | 甲 | release 投放语义，价值中等 |
| **P3（需决策）** | 阈值类 8 个 | 乙 | **依赖"是否引入规则引擎层"决策**，先不做 |

P1 全部做完，事件类型从 5 → 8，纯增益、零哲学代价。P2 做完到 10。P3 待决策。

## 四、实现要点（P1 + P2，集中在 sim_server.py）

### 4.1 `_diff_and_emit` 扩展结构

当前函数主体是"machines 段 + transfers 段"。扩展后建议分段清晰：

```python
def _diff_and_emit(self, frame: dict):
    prev = self._prev_frame
    if prev is None:
        self._prev_frame = frame
        return

    self._diff_machines(prev, frame)          # 已有：machine_start_op / machine_idle
    self._diff_transfers(prev, frame)          # 已有：transfer_started / transfer_completed
    # —— P1 新增 ——
    self._diff_jobs(prev, frame)               # job_completed / job_released
    self._diff_transfer_destination(prev, frame)  # transfer_destination_resolved
    # —— P2 新增 ——
    self._diff_agvs(prev, frame)               # agv_assigned / agv_freed

    self._prev_frame = frame
```

`episode_completed / episode_truncated` **不在 `_diff_and_emit` 里**——它们由 `_run_loop` 的终止判定（`sim_server.py:644`）直接 emit，不走 diff。

### 4.2 各段 diff 规则伪码

**`_diff_jobs`**（P1）：
```python
prev_done = {j["job_id"]: j["is_completed"] for j in prev.get("jobs", [])}
for j in frame.get("jobs", []):
    jid = j["job_id"]
    if not prev_done.get(jid, False) and j["is_completed"]:
        self._emit_event("job_completed",
                         f"Job{jid} 完成（{j['progress']['done']}/{j['progress']['total']} 工序）",
                         level="success")
# job_released: 跨 release 阈值，用 self._released_fired 集合去抖
```

**`_diff_transfer_destination`**（P1）：
```python
prev_dst = {t["task_id"]: t.get("destination") for t in prev.get("active_transfers", [])}
for t in frame.get("active_transfers", []):
    tid = t["task_id"]
    if prev_dst.get(tid) is None and t.get("destination") is not None:
        self._emit_event("transfer_destination_resolved",
                         f"Task{tid} 目标确定为 {t['destination']}",
                         level="info")
```

**`_diff_agvs`**（P2，需先扩 frame）：
```python

for i, (pt, ct) in enumerate(zip(prev["grid_state"]["tasks"],
                                 frame["grid_state"]["tasks"])):
    if pt is None and ct is not None:
        self._emit_event("agv_assigned", f"AGV{i} 接单 Task{ct}", level="info")
    elif pt is not None and ct is None:
        self._emit_event("agv_freed", f"AGV{i} 完成任务进入空闲", level="success")
```

## 六、机制重构：声明式 diff 引擎（替代 if/else 链）

### 6.0 当前实现的丑陋之处

`_diff_and_emit`（`sim_server.py:302-371`）的本质问题：**规则定义（"什么跃迁算事件"）与执行流程（for/if/emit）混在一个函数体里**。每加一条规则就要：
1. 在函数里加一段 `prev_xxx = {...}` 取视图
2. 加一个 `for ... if old ... if cur ...` 循环
3. 在循环里直接 `self._emit_event(...)`

函数会无限膨胀，规则之间混杂，测试只能整段跑，P3 阈值类加进来更是一团乱麻。

### 6.1 核心思想：规则即数据，执行器即循环

把"取视图 + 比对"两件事抽象成**规则对象**，每条规则只声明"我从 frame 里取什么、怎么比"，执行器统一负责"调规则 → 收 emission → 广播"。新增事件 = 注册一条规则，**不动执行器**。

```
frame ──► Rule.extract(frame) ──► view ──┐
                                         ├─► Rule.compare(prev_view, cur_view, ctx) ──► [Emission, ...]
frame ──► Rule.extract(frame) ──► view ──┘
```

### 6.2 抽象分层（自底向上）

#### (a) `Emission` ——事件产出值对象（纯数据）
```python
@dataclass
class Emission:
    type: str            # "machine_start_op" ...
    message: str
    level: str = "info"
    payload: dict | None = None   # 引擎相关业务字段，可选
```

#### (b) `DiffContext` ——本 episode 的累积状态（去抖/历史用）
```python
@dataclass
class DiffContext:
    step: int
    env_timeline: float
    fired: set[str] = field(default_factory=set)   # 已触发的事件 key，用于阈值去抖
    history: dict = field(default_factory=dict)     # 规则可挂自定义状态
```
阈值类规则用 `ctx.fired.add(key)` 确保同一异常只发一次；状态机类规则用不到。

#### (c) `Rule` ——抽象基类（两条方法）
```python
class Rule(Protocol):
    name: str
    def extract(self, frame: dict) -> Any: ...
    def compare(self, prev_view: Any, cur_view: Any, ctx: DiffContext) -> Iterable[Emission]: ...
```
- `extract`：从 frame 取**最小可比对视图**（dict/list/scalar）。这步隔离了"frame 形状变化"对比对逻辑的影响——frame 改字段时只动 extract，不动 compare。
- `compare`：纯函数，**输入两份视图 + ctx，输出 Emission 列表**。不接触 frame，不接触广播，可独立单测。

#### (d) 内置三种 Rule 子类（覆盖所有已知事件类型）

| 子类 | 适用 | 已知用户 |
|---|---|---|
| `StateTransitionRule` | 按键索引的状态机：`{key: state}` 的状态跃迁 | machine op、AGV task、op status、job is_completed |
| `SetMembershipRule` | 集合元素出现/消失 | active_transfers.task_id |
| `ThresholdRule` | 连续量跨阈值 | queue_length、wait_time、deadline、metric spike |

**StateTransitionRule** 声明式用法（覆盖现有 machines 段 + P2 AGV + job_completed）：
```python
StateTransitionRule(
    name="machine_op",
    extract=lambda f: {m["id"]: _op_key(m.get("current_op"))
                       for m in f.get("machines", [])},
    transitions={
        (None,      "*"):      lambda mid, s, d: Emission("machine_start_op",
                                              f"M{mid} 开始 Job{d[0]}-Op{d[1]}", "info"),
        ("*",       None):     lambda mid, s, d: Emission("machine_idle",
                                              f"M{mid} 完成加工，进入空闲", "success"),
        ("*",       "*diff*"): lambda mid, s, d: Emission("machine_start_op",
                                              f"M{mid} 切换至 Job{d[0]}-Op{d[1]}", "info"),
    },
    label=lambda k: f"M{k}",
)
```
- `"*"` 通配：任意非 None 值
- `"*diff*"`：old 与 cur 都非 None 且不同
- 这一条规则**完整等价于现在 `_diff_and_emit` 的 machines 段 25 行代码**

**SetMembershipRule**（覆盖 transfers 段）：
```python
SetMembershipRule(
    name="transfers",
    extract=lambda f: {t["task_id"]: t for t in f.get("active_transfers", [])},
    on_added=lambda tid, t: Emission("transfer_started",
        f"AGV 取货 Task{tid} (Job{t.get('job_id')}-Op{t.get('op_id')})", "info"),
    on_removed=lambda tid: Emission("transfer_completed",
        f"AGV 完成 Task{tid}", "success"),
)
```

**ThresholdRule**（P3 阈值类专属）：
```python
ThresholdRule(
    name="deadline_risk",
    extract=lambda f: [(j["job_id"], j["due"], j["is_completed"], j["progress"])
                       for j in f.get("jobs", []) if j.get("due") is not None],
    probe=lambda jid, due, done, prog, ctx:
        None if done else (
            Emission("deadline_risk", f"Job{jid} 接近 deadline {due}", "warning")
            if ctx.env_timeline > due * 0.8 and f"deadline:{jid}" not in ctx.fired
            and ctx.fired.add(f"deadline:{jid}") is None  # add 返回 None，串联短路赋值
            else None
        ),
)
```
- `probe` 对每个元素独立判定，返回 Emission 或 None
- 用 `ctx.fired` 去抖，同一 key 只发一次
- 阈值规则的"丑陋"被圈在 probe 闭包里，不污染 diff 主线

#### (e) `DiffEngine` ——执行器（10 行，永不修改）
```python
class DiffEngine:
    def __init__(self, rules: list[Rule]):
        self.rules = rules
        self._prev_views: dict[str, Any] = {}
        self.ctx = DiffContext(step=0, env_timeline=0.0)

    def process(self, frame: dict) -> list[Emission]:
        cur_views = {r.name: r.extract(frame) for r in self.rules}
        emissions = []
        for r in self.rules:
            prev = self._prev_views.get(r.name)
            if prev is not None:   # 首帧跳过，建基线
                emissions.extend(r.compare(prev, cur_views[r.name], self.ctx))
        self._prev_views = cur_views
        return emissions
```

`sim_server.py` 侧只剩两行：
```python
# init
self.diff_engine = DiffEngine(rules=build_default_rules())

# _run_loop 内（替换 _diff_and_emit 调用）
for em in self.diff_engine.process(frame):
    self._emit_event(em.type, em.message, em.level)
```

### 6.3 收益

| 维度 | if/else 版 | 声明式版 |
|---|---|---|
| 加一条事件 | 改 `_diff_and_emit` 函数体 | `RULES.append(...)` 一行 |
| 单测 | 需起整个 engine | `rule.compare(v1, v2, ctx)` 纯函数测 |

## 八、与 `0627static对齐难点.md` 的联动

- **难点 A（事件语义落差）**：本扩展把 level 扩到 `warning/error`，StaticFactory 的"设备告警/故障排除"两条事件可映射到 `type:"machine_anomaly"` + `level:warning/error`，比单纯 `narrative` 兜底更对齐容器化词汇表（前提是容器化侧也认这条 type——目前没有故障模型，可预留 type 名）。
