## 1. 模块①：Machine & Job 状态细化

### 1.1 现状
- `sim_server.py:459-489` `_build_frame` 当前 machine 字段只有 `{id, location, current_op:[job_id,op_id] | null}`
- machine 缺：op 在 JOB 中是第几道 / 共几道、op 已加工步数 / 预计步数、machine 队列长度、machine 自身状态
- **Job 完全没在 frame 里**：`_build_frame` 只输出 machines / active_transfers，没输出 jobs。但引擎侧 `pogema_env.jobs`（`grid_factory_env.py:241`）里 Job / Operation 字段齐全（见下），是 sim_server 没往外吐
- 引擎已有的 Job/Op 字段（`Utils/structure.py`）：
  - `Job`: `ops: List[Operation]`, `release`, `due`, `completion_time`, `is_completed`
  - `Operation`: `status`(PENDING/PROCESSING/FINISHED), `proc_time`, `arrive_machine_at`, `start_process_at`, `finish_process_at`, `wait_for_machine_time`, `assigned_machine`

### 1.2 frame 字段升级（sim_server.py 改造点）

**a) machine 字段**（`_build_frame` 输出每个 machine 改为）：
```json
{
  "id": 0, "location": [3,4],
  "status": "WORKING | IDLE | BLOCKED",
  "current_op": {
    "job_id": 3, "op_id": 2,
    "index_in_job": 2, "total_in_job": 4,
    "step_done": 5, "proc_time": 12
  },
  "queue_length": 2
}
```
- `current_op` 二元组废弃，统一成对象；`null` 表示空闲
- `step_done` / `proc_time` 从 `Operation` 对象取（`proc_time` 已有；`step_done` 由 `start_process_at` 与当前 step 差值算出）
- `index_in_job` / `total_in_job` 由 op 在 `job.ops` 中的索引 + `len(job.ops)` 得到
- `status` 由 `current_op` + 是否被 transfer 阻塞推断
- `queue_length` 从 job_solver 队列或 machine 前置 buffer 取（需确认字段位置）

**b) 新增顶层 `jobs` 字段**（`_build_frame` 输出新增）：
```json
"jobs": [
  {
    "job_id": 3,
    "release": 0.0,
    "due": null,
    "is_completed": false,
    "completion_time": -1.0,
    "progress": { "done": 2, "total": 4 },
    "ops": [
      {
        "op_id": 0,
        "status": "FINISHED",
        "proc_time": 12,
        "assigned_machine": 1,
        "arrive_machine_at": 5.0,
        "start_process_at": 5.0,
        "finish_process_at": 17.0,
        "wait_for_machine_time": 0.0
      },
      {
        "op_id": 1,
        "status": "PROCESSING",
        "proc_time": 8,
        "assigned_machine": 0,
        "arrive_machine_at": 17.0,
        "start_process_at": 17.0,
        "finish_process_at": -1,
        "wait_for_machine_time": 0.0
      },
      { "op_id": 2, "status": "PENDING", "proc_time": 10, "assigned_machine": null },
      { "op_id": 3, "status": "PENDING", "proc_time": 6, "assigned_machine": null }
    ]
  }
]
```
- 直接从 `pogema_env.jobs` 遍历生成，字段对齐 `Operation` / `Job` dataclass
- 大批量 Job 时可考虑只吐"未完成 + 最近完成"的子集，避免 frame 过大（待评审）

### 1.3 前端改造

**Machine 侧**（`stores/factory.js` / `components/FactoryVisualization/*`）：
- machine 卡片新增两条进度
  - 上条：op 步数进度（`step_done / proc_time`）
  - 下条：op 在 JOB 中的位置（`index_in_job / total_in_job`）
- 点击展开详情面板：current_job/op、queue、machine_status
- `current_op == null` 降级显示，不崩

**Job 侧**（新增 Job 列表 / 看板）：
- 在 FactoryTabsPanel 新增一个 tab（或在 events / metrics tab 内嵌），展示当前 frame 的 `jobs` 数组
- 每个 Job 卡片显示：
  - `job_id` + 总进度条（`progress.done / progress.total`）
  - 状态标签：`is_completed` → 已完成；否则 `done/total` 道完成
  - 交期 `due` 与 `completion_time`（如已完工）/ 当前 step 对比，超期高亮
  - op 时间轴：每个 op 一行，按 PENDING / PROCESSING / FINISHED 着色，显示 `proc_time` / `assigned_machine` / `wait_for_machine_time`
- 点击单条 Job 展开全部 ops 明细（arrive/start/finish 时间戳、等待时长）
- 与 machine 侧联动：点击 machine 的 `current_op` 高亮对应 Job；点击 Job 中 PROCESSING 的 op 高亮对应 machine

**通用约束**：
- 进度口径：**按工序步数计数**（已确认），不做时间估算
- `current_op == null` / `jobs` 字段缺失时降级显示，不崩
- 颜色规范：自行设计（machine 进行中蓝 / 阻塞橙 / 即将超时红；op PENDING 灰 / PROCESSING 蓝 / FINISHED 绿）

## 2. 模块②：在线自定义绘图（大禹迁移）

### 2.1 核心问题
`MetricsPanel.vue` 现状是反例：
- `SIM_METRICS_META`（行 170-184）写死 13 个指标
- 折线图（行 229-258）写死 machine_utilization / agv_loaded_utilization 两条线
- 热力图（行 295-333）写死 occupancy，用 div grid 不用 ECharts

新增指标 / 换图表样式都要改组件代码，违反师兄三条铁律。

### 2.2 三层落地（仅 DockerFactory）

**数据层 Hook**（`frontend/src/hooks/metrics/*.js`）
```js
// hooks/metrics/machine_util.js
export default {
  id: 'machine_util', label: '机器利用率', unit: '%',
  series: (monitor) => monitor.metricsTimeline.map(p => ({
    x: p.step, y: (p.metrics?.machine_utilization ?? 0) * 100,
  })),
}
```
- Hook 是纯函数，只从 `monitor.js` 抽 `{x, y}` 序列
- 第一批 Hook 来源就是 `SIM_METRICS_META` 那 13 个指标 + episode 终态（makespan、occupancy）
- 新增指标只加 Hook 文件 + 注册，不动模板

**模板层**（`frontend/src/components/charts/*.vue`）
- `LineChart` / `BarChart` / `StackedAreaChart` / `HeatmapChart`
- 每个组件**只接收标准 series**，不知道指标名
- 把现有 `createMiniOption/createBigOption`（行 360-395）拆进来
- occupancy div grid 换 ECharts heatmap

**配置层**（`config/dashboards/docker_factory.json`）
```json
{
  "id": "docker_factory_default",
  "cards": [
    {"hook":"machine_util","template":"line","yMax":100},
    {"hook":"agv_loaded","template":"line","overlay":["agv_busy"]},
    {"hook":"swap_conflict","template":"bar"},
    {"hook":"occupancy","template":"heatmap"}
  ]
}
```
- 用户改配置切换看板组合，零代码改动
- 静态优先（JSON 文件驱动），二期再做前端拖拽编辑器

### 2.3 现有代码改造路径
1. 新建 `hooks/metrics/` 目录，逐个迁出 `SIM_METRICS_META` 指标
2. 新建 `components/charts/`，把 `MetricsPanel.vue` 的 ECharts option 抽过去
3. `MetricsPanel.vue` 改成读 `config/dashboards/docker_factory.json` 渲染卡片
4. 旧的 PacketFactory 分支保留（`v-else` 那一段不动），只重写 DockerFactory 分支

## 3. 模块③：离线日志分析（华佗迁移）

### 3.1 关键洞察
`sim_server.py:302-366` 的 `_diff_and_emit` **已经在派生业务事件**（machine_start_op / machine_idle / transfer_started / transfer_completed）推给 `/stream/events`。

→ **sim_server 改造最小切口**：给 `_emit_event` 再加一个 JSONL sink，不用重新实现事件检测。每跑一次 episode 产出一个 `sky_logs/episode_{run_id}.jsonl`。

### 3.2 数据流
```
sim_server 跑批 → sky_logs/episode_{run_id}.jsonl
                    ↓ POST /api/logs/upload
                run_id = xxx
                    ↓ POST /api/logs/{run_id}/analyze
            离线 Hook 计算 metrics_dict
                    ↓ 走模块②同一套 Chart 模板
            views/analysis/
```

### 3.3 离线 Hook（契约对齐在线）
```js
// hooks/offline/machine_util.js — 与在线同名 Hook 接口一致
export default {
  id: 'machine_util',
  series: (parsedLog) => parsedLog.steps.map(p => ({
    x: p.step, y: p.metrics.machine_utilization * 100,
  })),
}
```
- 前端配置层完全复用（看板 JSON 只写 `hook:"machine_util"`，由数据源类型决定调在线还是离线版本）
- 这就是"在线离线共用渲染内核"在天工的落地方式

### 3.4 后端新增（`application/backend/logs/`）
- `POST /api/logs/upload` — 接收 `.jsonl/.zip`，校验 schema，落 `analysis_runs/`
- `POST /api/logs/{run_id}/analyze` — 解析日志，跑离线 Hook 计算指标字典
- `POST /api/logs/export` — 按 run_id / 时间窗筛选打包下载
- 注册到 `core/RouteRegistry.py`

### 3.5 离线交互
师兄强调的时间拖拽 / 区间筛选 = ECharts `dataZoom`，模板层统一加上（在线离线都受益）。

## 4. 模块三（批处理）与模块③的关系

`需求整理_2026Q2.md` 的批处理模块跑完一批次的产物就是 N 份 `episode_*.jsonl`，**直接喂给模块③的离线分析**。所以：
- 模块③（离线分析）是批处理的下游消费者
- 批处理本身复用 `DockerProxy._compose()` + `ALGORITHM_MAP`，不另起编排
- 批处理新增 `application/backend/batch/` + 前端 `views/batch/`，跟 DockerFactory 在线模式共用 engine 容器和 sim_server

## 5. 设计纪律（继承自师兄三条铁律）

1. 任何指标渲染需求：**先写 Hook、再选模板、最后改配置**。不允许直接在组件里写死指标名 + ECharts option
2. 新增图表样式 = 加模板组件，所有指标共用，不 fork 老组件
3. 在线和离线的 Chart 模板必须共用同一份代码
