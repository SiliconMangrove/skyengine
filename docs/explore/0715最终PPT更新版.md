**P2 · 问题与现有方案不足**
- FMS 调度算法研究的核心痛点：
  - **柔性定义不明确**：不同文章对 FMS 的理解差异很大，调度算法和路由算法之间无法横向比较。
  - **算法-仿真-可视化割裂**：同一个算法换个人跑、换个数据集，结果对不上；研究者大量时间花在重写仿真器而非算法本身。
  - **复现难、可比性差**：现有平台要么不支持在线响应（异常/插单），要么不开放，要么不可工业化。
- 现有竞品：
    - **POGEMA**：纯多智能体路由，每轮任务调度由系统随机生成，不含真实语义。
    - **JOBSHOP-LIB**：纯任务调度，每轮的路由默认完成，无法建模路由拥塞等真实现象。
- 收尾一句话：**"我们要做的，是把这两端连同中间的指派层，放进同一个可复现、可插拔、可响应的数字孪生里。"**
- 讲述要点：1–1.5 分钟讲完，让听众认同"这个问题值得解决"。

**P3 · 核心假设（Thesis Page）**

> 这 3 个假设是后面所有内容的"心智锚点"，P13 反思页会逐条回应。

**H1 · 业务领域三层解耦**（柔性建模能力的基础）
- **任务调度层**：把 JOB 分解为带候选机器的工序序列
- **指派层**：把工序任务交付给具体的 AGV 执行
- **路由层**：AGV 在网格中推进任务的最优路径
- 三层均可独立替换算法 —— 这是"算法可插拔"的能力基础

**H2 · 系统架构三仓库物理分离**（异构栈零侵入集成的基础）
- **系统层** `grouppro/skyengine`：FastAPI 后端 + Vue 前端 + FactoryProxy 抽象
- **引擎层** `finalpro/SkyEngine`：仿真推进核心（sim_server）
- **算法层** `finalpro/MAPF` + `finalpro/FJSP`：两类求解器独立仓库
- 三层通过 DockerProxy + docker compose 桥接 —— 服务层零算法、算法层零业务

**H3 · SSE 三流 + 事件信封**（实时与离线统一的基础）
- state 流 / metrics 流 / events 流 三流并行
- 同一套事件信封同时支撑：实时可视化推送 + JSONL 离线归档分析
- 避免了"实时协议 + 离线协议"双轨制的复杂度

**P4 · 里程碑时间轴**

> 每个节点用 **效果 / 问题 / 决策** 三段式；技术线和认知线双线并行。

- **节点 1：2025.5 — 仿真内核诞生**
  - **效果**：从零搭建 AGV / Machine / Job / Operation 四元组领域模型，`env_step(actions, step_time)` 作为统一仿真接口，引入标准 FJSP benchmark。
  - **问题**：仓库空置两月，对 FMS 缺乏系统认知；不存在可直接复用的现成方案。
  - **决策**：用标准 benchmark 而非自造数据；同步引入 GreedyAgent 验证环境可运行。
  - **长期影响**：`env_step` 接口贯穿后续所有版本；四元组模型至今未变。

- **节点 2：2025.7 — 前后端分离 + Web 化** ⭐ 技术转折点
  - **效果**：推翻 pygame 桌面，全面转向 FastAPI + Vue Web 架构；引入不确定性建模与异常事件机制。
  - **问题**：单体程序科研侧与业务侧混杂；PNG 序列帧展示状态不美观；无路由层建模，只能解决任务调度问题。
  - **决策**：采用分体式架构，展示层作为业务侧（grouppro），强化学习引擎作为科研侧（finalpro），分开维护；同时引入路由层建模、pogema 作为网格基座。原 PacketFactory 由杭业晟同学长期维护。
  - **长期影响**：奠定了"系统层 / 引擎层"两仓库分离的格局，是 H2 假设的第一次落地。

- **节点 3：2025.11 — 三层分层建模 + 多工厂启动融合** ⭐ 反过度设计的克制决策
  - **效果**：GridFactory 的任务调度、指派、AGV 路由三层建模基本完成；前端登录页可切换工厂；Assigner/JobSolver/RouteSolver 四层调度抽象落地。
  - **问题**：多种工厂在启动时各自指定，无法体现"同一系统下的多引擎"；PacketFactory 与 GridFactory 引擎异构，强行塞进单 server 两边都扭曲。
  - **决策**：选择"先双 server 并存、后 Proxy 抽象"——先让两种引擎各自跑通，把统一接口留给下一阶段。事后证明这条路径是正确的：没有这次实践素材，2026.3 的 BaseFactoryProxy 协议就无法定义得足够通用。
  - **长期影响**：为 Proxy 化提供了实战素材；这是 SkyEngine "算法可插拔"假设（H1）的第一次局部验证。

- **节点 4：2026.3 — v1.0.0 公开发布 + FactoryProxy 抽象诞生** ⭐ 平台化真正起点
  - **效果**：完成翼辉相关案例并交付给南邮同学；多工厂通过统一 Proxy 接口实现基本一致的操作逻辑；StaticFactory / PacketFactory / GridFactory 三类 Proxy 全部接入。
  - **问题**：算法与引擎层强耦合，调试困难；多引擎各自为政、无统一接口；要开源发布必须有清晰架构。
  - **决策**：拆分引擎层与算法层，统一用 docker 打包求解器，支持异构算法环境；引入 `BaseFactoryProxy` + `ProxyFactory` + `RouteRegistry` 三件套，Proxy 协议定义 init / start / pause / reset / stream 等统一接口。
  - **长期影响**：接入新引擎成本从"写一整套 server"降到"实现一个 Proxy 类"；SSE 取代轮询成为前后端通信主通道；CLAUDE.md 引入标志 AI 辅助开发时代。

- **节点 5：2026.5 — 完整容器化改造** ⭐ 假设 H2 的终极验证
  - **效果**：实现系统层、引擎层、算法层的全链路容器化；DockerProxy 通过 docker compose 拉起 engine + fjsp + mapf 三容器，零加工透传 SSE 流；接入 finalpro/SkyEngine 作为引擎层、finalpro/MAPF + finalpro/FJSP 作为算法层。
  - **问题**：真实仿真引擎在独立项目里，如何接入？算法栈异构（PSO/DE/DRL 与 A*/MAPF-GPT 环境依赖完全不同），如何统一管理？
  - **决策**：DockerProxy 仅做"透传 + 生命周期管理"，引擎和算法解耦交给容器；算法镜像可热替换，配置即可切换求解器。
  - **长期影响**：验证了 Proxy 抽象的复利效应——节点 2 投入的架构资本在这一刻连本带利回来；前端零修改接入完全不同的仿真引擎。

- **节点 6：2026.7 — 完善项目 + 最终交付**
  - **效果**：完成迭代（异常事件、指标计算、可视化细节、批实验支持、FJSP spec v2、JSONL 归档）；稳定性测试；最终文档与 demo 演示。
  - **问题**：包括与师弟讨论的思考：推理时延与 step 时长的对齐、不同 OBS 的支持、FJSP 插单格式对齐标准。
  - **决策**：形成完善的系统文档（docs/explore 体系），后续持续跟进。
  - **长期影响**：项目进入"可移交给师弟"的成熟态。

**P6 · FactoryProxy + DockerProxy（技术深度专题）**

**FactoryProxy 抽象边界**
- Proxy 类声明了 `initialize / start / pause / reset / stop / cleanup` 生命周期方法，加上 `set_config / set_algorithm / 三流接口`。不同工厂实现统一接口，前端代码零修改即可切换。
- 当前已接入：
  - **StaticFactory**：纯前端剧本，调试用 mock
  - **PacketFactory**：进程内电池产线仿真（杭业晟维护）
  - **GridFactory**：pogema 网格工厂，分容器化与非容器化两版
- 接入成本对比：
  - 老路径（v1.0 前）：写一整套 server + 路由 + 通信 ≈ 数周
  - 新路径（Proxy 抽象后）：实现一个 Proxy 类 ≈ 数天

**DockerProxy 容器化工厂管理**
- **两阶段启动**：
  - Phase 1：`docker compose up engine` → 等待 engine 健康检查
  - Phase 2：`docker compose up mapf fjsp` → POST `/sim/create` 配置 → POST `/sim/play` 启动
- **SSE 透传机制**：引擎容器产生 state/metrics/events 三流 SSE，DockerProxy 用 httpx 流式读取后零加工转发给前端。Proxy 不缓冲、不加工，性能与直连几乎等价。
- **算法镜像热替换**：通过 docker compose 环境变量切换镜像（`MAPF_IMAGE` / `FJSP_IMAGE`），用户在前端下拉选算法 → 后端 set_algorithm → 下次启动时换镜像。算法栈零侵入接入。
- 讲述要点：这是全场第二重要的一页。如果听众只记一件事，要让他们记住"**算法栈异构怎么办 —— 用容器 + 透传 Proxy 桥接**"。

**P7 · 动态响应能力：异常注入 + 在线插单**

> 这两个能力不是"两个技术点合并"，而是**同一个能力的两个维度**——让仿真从"放映机"变成"交互沙盘"。

**统一框架：动态响应能力**
- 传统仿真平台只能跑确定性计划。真实制造系统的核心挑战是**不确定性**：机器故障、AGV 故障、紧急插单。SkyEngine 的引擎层和服务层共同提供动态响应能力。

**维度一 · 引擎层 · 异常事件注入**
- 通常情况下异常会造成各个组件的下线（machine / AGV / job）。
- 通过独立的 **Injector 组件**生成掩码向量，不同掩码代表当前 id 的组件执行什么异常事件（machine fail / AGV fail / env pause / job cancel）。
- 掩码向量作为一种**额外的特征向量**，可顺畅接入后续强化学习模型——这是"异常可学习"的关键设计。

**维度二 · 服务层 · 在线插单（FJSP spec v2）**
- 通常情况下不同的 FJSP 求解器与 MAPF 求解器均可支持在线求解。
- 用户在前端 JobInsertPanel 上传标准 FJSP 实例文件（如 `J10P5M6.json`）→ 前端解析预览（机器数匹配校验、候选分布统计）→ POST 到 DockerProxy → 透传到引擎 `/sim/insert_jobs`。
- **批量 partial 语义**：valid job 入队、invalid job 单独列出拒绝原因，前端 UI 即时反馈。
- **spec v1 → v2 演进**：v1 是业务接口式（自定义字段），v2 直接对齐 FJSP 学术标准实例格式（`{machines, jobs, extensions}`），让平台可以直接消费学术 benchmark 数据集——这是与 POGEMA/JOBSHOP-LIB 拉开差距的关键点。

**共同的支撑 · 事件信封**
- 异常事件和插单事件都通过统一的 canonical envelope 进入 events 流
- 同一套信封同时推送前端 EventPanel + 写入 JSONL 归档
- 验证了 H3 假设：实时与离线共用一个协议

- 讲述要点：先讲"动态响应"这个统一概念（30 秒），再分两个维度展开（各 1.5 分钟），最后收一句"两个能力共用同一套事件信封，是 H3 假设的实战验证"。

**P12 · 解决的实际问题 + 量化收益**

**对应 P2 痛点的逐条回应**

| P2 痛点 | 天工的解决方案 |
|---|---|
| 柔性定义不明确、横向比较难 | H1 明确三层解耦（任务调度/指派/路由），每层独立可替换 → **可在同一平台上横向比较调度策略与路由策略** |
| 算法-仿真-可视化割裂 | H2 三仓库物理分离 + DockerProxy 桥接 → **算法研究者只写算法、不碰仿真代码** |
| 复现难、可比性差 | 容器化部署 + 标准 FJSP benchmark + JSONL 归档 → **任何人 5 分钟复现任意一次实验** |
| 现有平台不支持动态响应 | 异常注入 + 在线插单（FJSP spec v2 对齐学术标准） → **可研究响应式调度** |

**量化收益**
- **接入工厂**：3 类（StaticFactory / PacketFactory / GridFactory）
- **接入算法**：FJSP × 4（PSO/DE/DRL/BEST）+ MAPF × 2（A*/MAPF-GPT）+ Assigner × 9 = **共 15 种算法可插拔组合**

**P13 · 反思 + 未来方向（合并页，灵魂页不可省）**

> 这是整场 PPT 最重的一页，语速放慢，真诚讲，不要包装成"一切都很顺利"。

**对 H1/H2/H3 假设的回顾**
- ✅ **H1（三层解耦）成立**：任务调度、指派、路由三层独立替换在 GridFactory 上充分验证，Assigner/JobSolver/RouteSolver 抽象贯穿全项目。
- ✅ **H2（三仓库分离）成立且超预期**：DockerProxy 模式不仅接入了 finalpro 引擎层，更让"算法层独立仓库"成为可能。原以为只是工程隔离，实际上让算法迭代节奏与系统解耦——算法仓库的 PSO 团队可以独立于路由团队发布镜像。
- ⚠️ **H3（SSE 流 + 事件信封）部分成立**：实时推送用 SSE 完全够用；但"同一套协议同时支撑离线"是部分成立的——离线分析需要的字段比实时展示多（如完整的 payload、上下文 step、job_id 等），最终做了"事件信封 + JSONL 补充字段"的妥协方案。**这本质上是 H3 的过度乐观——实时和离线的诉求本就不完全相同。**

**过度设计 / 欠设计**
- 🟢 **不算过度设计的"过度设计"**：早期（2025.5）只有一种工厂时就投入做 Registry/Callback 抽象，当时看起来过度。但后续 5 次大变更每次都只需要"新增一层"——这是典型的架构投资复利。
- 🟡 **真实的欠设计**：FJSP 插单 spec v1 一开始没对齐学术标准 FJSP 实例格式，自定义了一套业务字段，导致用户上传标准 benchmark 文件时无法直接消费，最终做了 v2 重构（v1 保留 alias 兼容 6 个月）。教训：**做接口设计前先调研目标领域的现有标准，不要凭业务直觉造轮子。**
- 🟡 **方法论反思 · 抽象时机的判断**：节点 2（2025.5 末）早抽象 Registry，节点 3（2025.11）克制没抽象 Proxy——这两种相反的决策都正确，因为时机不同。**判断"什么时候抽象、什么时候不抽象"是这一年想明白的最重要的事。**

**未来方向**（1–2 个最值得继续的方向）
1. **推理时延与 step 时长的对齐建模**：当前 step 时长是仿真虚拟时间，与算法推理时延解耦。未来需要把推理时延锚定到现实时间，研究"算法思考慢一拍"对调度质量的影响——这是把仿真推向真实工程部署的关键一步。
2. **多 OBS 支持**：当前 Observation 是固定的；面向强化学习训练，需要支持多视角 OBS（全局/局部/混合）切换，让平台真正成为 RL 训练基础设施。

- 讲述要点：反思页 3 分钟，未来方向 1 分钟，合并 4 分钟。
