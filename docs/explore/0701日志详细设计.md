## 0. 已收敛决策清单

| # | 议题 | 决策 |
|---|---|---|
| C1 | 持久化介质 | 后端文件系统 `dataset/run/`，扁平结构 |
| C2 | Run id = 文件名 | `<factory_id>__<algorithm>__<timestamp>.json` |
| C3 | factory_id 规范 | StaticFactory 统一用 `static_factory`（含 `northeast_center` alias） |
| C4 | algorithm 字段 | 照填用户选的字符串，`+` 替换为 `_`（如 `fifo_pibt_random`）；为空填 `none` |
| C5 | 落盘逻辑归属 | **独立 AnalysisService**（`application/backend/analysis/`），工厂无关 |
| C6 | 自动落盘策略 | **每次 episode 结束都写**，不去重 |
| C7 | 去重 | **不做**。理由：StaticFactory 收益小、容器化根本用不上、副作用大 |
| C8 | 导入策略 | **导入即入库**，写盘到 `dataset/run/`，特殊命名区分 |
| C9 | 导入命名 | 文件名 `imported__<stem>__<timestamp>.json`；Run 内部 `source: 'imported'`；保留 `original_filename` |
| C10 | 列表 API | 分两层：`GET /analysis/runs`（轻，仅 summary）+ `GET /analysis/runs/{id}`（重，含三流） |
| C11 | 列表性能 | 短期全量扫描目录；Run 上几百再上 `index.json` manifest |
| C12 | schema_version | **必做**，文件首字段 `"schema_version": "1.0"` |
| C13 | summary 协议 | **永远派生**，importRun 时一律重算，外部 summary 字段忽略 |
| C14 | config_snapshot | **必做**，Run 自包含可复现单元 |
| C15 | 导出 vs 下载 | "下载"取代"导出"——已存档 Run 直接 `GET /analysis/runs/{id}` 返回文件 |
| C16 | 删除 | 走 `DELETE /analysis/runs/{id}`，不做回收站 |
| C17 | 跨会话 id 唯一性 | 文件名即 id（带秒级时间戳）；并发同秒撞名时后端追加短 hash 后缀 |

## 1. Run 文件命名规范（C2/C3/C4）

### 1.1 格式

```
<factory_id>__<algorithm>__<YYYYMMDD_HHMMSS>.json
```

**示例**：
```
static_factory__fifo_pibt_random__20260702_143501.json
packet_factory__spt_cbs_greedy__20260702_150023.json
```

### 1.2 字段规则

| 字段 | 来源 | 规范化 |
|---|---|---|
| `factory_id` | `meta.factory_id` 或 `factoryStore.selectedFactoryId` | lowercase；alias 收敛（`northeast_center` → `static_factory`） |
| `algorithm` | `meta.algorithm`（如 `fifo+pibt+random`） | `+` → `_`；空值填 `none`；其他非 `[a-z0-9_-]` 字符替为 `_` |
| `timestamp` | 后端落盘时刻 | 本地时间 `YYYYMMDD_HHMMSS` |

### 1.3 撞名处理

秒级时间戳理论上同秒会撞（多标签并发跑）。后端落盘前 `os.path.exists` 检查：
- 撞名 → 文件名末尾追加 `__<4位短 hash>`（取自内容 sha256 前 4 字节十六进制）
- 例：`static_factory__fifo_pibt_random__20260702_143501__a3f2.json`

### 1.4 导入样本命名（C9）

```
imported__<original_stem>__<YYYYMMDD_HHMMSS>.json
```

- `<original_stem>`：用户上传文件的主干（去扩展名），规范化为 `[a-z0-9_-]`
- 太长（>40 char）截断
- 例：用户上传 `my_run.json` → `imported__my_run__20260702_150012.json`

## 2. Run Schema（C12/C13/C14 落地）

```jsonc
{
  "schema_version": "1.0",          // 必填，文件首字段
  "id": "static_factory__fifo_pibt_random__20260702_143501",
  "source": "archive",              // "archive" (本地产出) | "imported" (用户上传)
  "original_filename": null,        // 仅 imported 有值，记录用户上传的原文件名
  "factory_id": "static_factory",
  "algorithm": "fifo+pibt+random",
  "created_at": "2026-07-02T14:35:01.123Z",  // ISO 8601 UTC
  "started_at": "T+0s",             // 来自首帧 env_timeline/timestamp
  "finished_at": "T+54s",           // 来自末帧
  "total_steps": 55,
  "config_snapshot": {              // C14 必填，自包含可复现
    "factory_layout": {
      "machines": [                 // 来自剧本常量 / 首帧 machines 投影
        { "id": "M1", "location": [8, 13] },
        { "id": "M2", "location": [11, 13] },
        { "id": "M3", "location": [14, 13] }
      ],
      "agv_count": 1
    },
    "job_spec": [                   // frame.jobs 的工艺路线投影
      { "job_id": 1, "release": 0, "due": 20,
        "ops": [{ "op_id": 0, "proc_time": 5, "assigned_machine": 1 }, ...] }
    ],
    "algorithm_params": null        // StaticFactory 留 null，容器化侧补
  },
  "frames":          [ /* 1.1 Frame[] */ ],
  "metricsTimeline": [ /* 1.2 Metrics[] */ ],
  "events":          [ /* 1.3 Event[] */ ],
  "summary": {                     // C13 永远派生，外部值忽略
    "total_steps": 55,
    "completed_jobs": 2, "total_jobs": 3,
    "avg_utilization": 0.42,
    "event_total": 6,
    "event_counts": { "sim_started": 1, "narrative": 4, "episode_completed": 1 }
  }
}
```

**字段优先级**：`schema_version` 必须第一（便于人工查看 + 未来按版本迁移）。

## 3. 后端 AnalysisService（C5/C10）

### 3.1 目录结构

```
application/backend/analysis/
  __init__.py
  service.py      # 纯逻辑，无 FastAPI 依赖
  routes.py       # FastAPI APIRouter，薄封装
```

仿照 `application/backend/history/` 的模式（工厂无关，`app.include_router` 顶层注册）。

### 3.2 service.py 接口

```python
class AnalysisService:
    def __init__(self, archive_dir: Path):
        self.archive_dir = archive_dir  # dataset/run/

    def list_runs(self) -> list[dict]:
        """扫描目录，返回所有 Run 的轻量元信息（不含三流）。
        每个 dict = {id, source, factory_id, algorithm, created_at, total_steps, summary}
        容错：损坏文件跳过 + 记 warning，不让单个坏文件炸整个列表。
        """

    def get_run(self, run_id: str) -> dict | None:
        """读完整 Run JSON。run_id 即文件名（无 .json 后缀）。
        路径校验：禁止 .. / 绝对路径，只能访问 archive_dir 下。"""

    def save_run(self, payload: dict, source: str = "archive") -> dict:
        """落盘一个 Run。
        - 补 schema_version / source / created_at
        - 重算 summary（忽略 payload 自带）
        - 生成规范化文件名，撞名追加 hash
        - 原子写入（tmp 文件 + os.replace）
        - 返回 {id, filename, path}
        """

    def delete_run(self, run_id: str) -> bool:
        """删除 Run 文件。返回是否删除成功。"""

    @staticmethod
    def _normalize_filename(factory_id: str, algorithm: str, source: str,
                             original_stem: str | None = None) -> str:
        """生成规范化文件名 stem（不含 .json）。"""

    @staticmethod
    def _compute_summary(frames: list, metrics_timeline: list, events: list) -> dict:
        """与前端 computeSummary 同形，对外部 JSON 容错。"""
```

### 3.3 routes.py

```python
router = APIRouter(prefix="/analysis", tags=["analysis"])

@router.get("/runs")           # 列表（轻）
@router.get("/runs/{run_id}")  # 详情（重）
@router.post("/runs")          # 落盘（body = Run JSON）
@router.delete("/runs/{run_id}")
```

**注册位置**：`server.py` 顶层 `app.include_router(analysis_router)`，与 `history_router` 同级，不依赖工厂切换。

**保留**：`StaticFactoryProxy.py` 里的 `/analysis/export` mock 路由暂留（前端旧调用未清干净前不删），新代码全走 `/analysis/runs/*`。

## 4. 前端 analysisLog.js 改造（C10/C12/C13/C14）

### 4.1 角色转变

- **之前**：`runs[]` 是内存数据源，finalize 直接 unshift
- **之后**：`runs[]` 是 `dataset/run/` 的内存镜像，所有写入走后端 API

### 4.2 公开接口

```js
// 状态
analysisLog.runs              // Run[]（镜像）
analysisLog.totalRuns
analysisLog.panelOpen
analysisLog.initialRunId
analysisLog.loading           // 列表加载中
analysisLog.currentDetail     // 详情缓存（按需加载）

// 浮层
openPanel(runId?)             // 自动调 listRuns()
closePanel()

// 后端交互
async listRuns()              // GET /analysis/runs → 更新 runs[]
async loadRun(id)             // GET /analysis/runs/{id} → currentDetail
async finalizeFromStores(factoryStore, monitorStore, meta)
                              // → POST /analysis/runs → listRuns 刷新
async importRun(payload, originalFilename)
                              // → POST /analysis/runs (source=imported)
async deleteRun(id)           // → DELETE /analysis/runs/{id}

// 派生
getRunById(id)                // 从内存 runs[] 查
computeSummary(frames, metrics, events)  // 与后端 _compute_summary 同形
```

### 4.3 finalizeFromStores 改造

```js
async finalizeFromStores(factoryStore, monitorStore, meta) {
  // 1. 深拷贝三流
  // 2. 构造 config_snapshot（StaticFactory 从剧本常量 + jobs 投影）
  // 3. 重算 summary
  // 4. 组装 Run payload（含 schema_version）
  // 5. POST /analysis/runs
  // 6. 成功后 listRuns() 刷新镜像
  // 7. return run（含后端回填的 id）
}
```

**注意**：返回值变成 Promise。`useSimulationConfig.snapshotToAnalysisLog` 改 `await`。

### 4.4 config_snapshot 构造（StaticFactory）

从 `fullSystemTest.js` 导出 `MACHINE_LOCATIONS` 和 `JOB_SCRIPT`：

```js
// fullSystemTest.js 新增导出
export const STATIC_FACTORY_CONFIG = {
  factory_layout: {
    machines: Object.entries(MACHINE_LOCATIONS).map(([id, location]) => ({ id, location })),
    agv_count: 1,
  },
  job_spec: JOB_SCRIPT.map(j => ({
    job_id: j.job_id, release: j.release, due: j.due,
    ops: j.ops.map(o => ({
      op_id: o.op_id, proc_time: o.proc_time, assigned_machine: o.assigned_machine,
    })),
  })),
}
```

`analysisLog.finalizeFromStores` 直接 import 用。容器化通路接入时，从首帧 machines 与 frame.jobs 投影出同形数据。

## 5. AnalysisOverlayPanel.vue 改造

### 5.1 列表加载

- `onMounted` → `analysisLog.listRuns()`
- `loading` 状态期间显示骨架屏
- 失败显示重试按钮

### 5.2 启用上传按钮

```html
<button class="ao-btn primary" @click="handleUploadLog">⬆ 上传日志分析</button>
<input ref="fileInputRef" type="file" accept=".json" hidden @change="onFilePicked" />
```

```js
async function handleUploadLog() {
  fileInputRef.value?.click()
}
async function onFilePicked(e) {
  const file = e.target.files?.[0]
  if (!file) return
  try {
    const text = await file.text()
    const payload = JSON.parse(text)
    const run = await analysisLog.importRun(payload, file.name)
    ElMessage.success(`已导入：${run.id}`)
    openDetail(run.id)
  } catch (err) {
    ElMessage.error(`导入失败：${err.message}`)
  } finally {
    e.target.value = ''  // 允许重选同文件
  }
}
```

### 5.3 卡片视觉区分

| source | 图标 | 角标 |
|---|---|---|
| `archive`（本地产出） | 📦 | 工厂 tag + 算法 tag |
| `imported`（用户上传） | 📁 | "导入" 灰色标签 + original_filename tooltip |

### 5.4 详情按需加载

进详情视图时 `await analysisLog.loadRun(id)`（详情数据不一定在列表镜像里，避免列表 API 传三流过重）。

## 6. 数据流（修订版）

```
┌─ StaticFactory 剧本 / 容器化 sim_server ─┐
│   每步推三个 store（流式累积）            │
└─────────────────┬───────────────────────┘
                  │ episode 结束
                  ▼
   finalizeFromStores (async)
     · 深拷贝三流
     · 构造 config_snapshot
     · 重算 summary
     · 组装 Run（schema_version=1.0）
                  │
                  │ POST /analysis/runs
                  ▼
   后端 AnalysisService.save_run
     · 校验 + 补字段
     · 规范化文件名
     · 原子写入 dataset/run/<id>.json
                  │
                  │ 返回 {id, filename}
                  ▼
   前端 listRuns() 刷新镜像 → UI 更新

── 用户点详情 ──→ GET /analysis/runs/{id} → currentDetail
── 用户点删除 ──→ DELETE /analysis/runs/{id} → listRuns() 刷新
── 用户点上传 ──→ 本地 file → JSON.parse → POST /analysis/runs (source=imported)
── 用户点下载 ──→ GET /analysis/runs/{id} → 浏览器下载
```
