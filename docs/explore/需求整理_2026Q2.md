## 0. 总体目标与适用范围

天工（SkyEngine）一期已打通"仿真-调度-可视化"基础链路。二期聚焦三件事：

1. **前端可视化粒度细化**：machine 展示当前正在处理的 operation 进度
2. **日志导出与分析模块**：离线日志的导出 / 上传 / 分析，绘图模板与指标解耦
3. **批处理模块**：一键调用 `finalpro/SkyEngine` 的实验编排，前端触发、后端拉起容器、回传结果

**重要范围说明（本次需求只覆盖以下两个工厂，不动其它）**：

| 工厂              | 前端入口                                                         | 后端实现                                                                                                                           | 形态                                                                                           |
| ----------------- | ---------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **StaticFactory** | `application/frontend/src/views/factory/StaticFactoryManage.vue` | `application/backend/core/StaticFactoryProxy.py`（skyengine 后端进程内 mock）                                                      | **无外部引擎**，mock 数据由 Python 内生成，AGV 轨迹 55 步硬编码                                |
| **DockerFactory** | `application/frontend/src/views/factory/DockerFactoryManage.vue` | `application/backend/core/DockerProxy.py`（透传） + `/data1/home/wuhao/project/finalpro/SkyEngine/sim_server.py`（引擎容器 :8080） | **真实仿真**，DockerProxy 通过 docker compose 拉起 engine + mapf + fjsp 三容器，SSE 零队列透传 |

不在本次范围：`packet_factory`、`grid_factory`（joint_sim 版本）以及未落地的 `southwest_logistics` 等其它工厂。所有改动只落到上述两个工厂对应的代码路径上。

**优先级**：模块1（前端细化）→ 模块3（批处理）→ 模块2（日志分析）。理由见各模块章节。

## 1. 模块一：Machine Operation 进度条

### 1.1 背景与现状盘点

当前两个工厂对 machine 的展示粒度都很粗：

**StaticFactory 现状**（`StaticFactoryProxy.py:98-131` `generate_machine_states`）：
```json
"M1": { "id": "M1", "status": "WORKING", "load": 60 }
```
有 `status`（IDLE/WORKING/BROKEN）和 `load`，**但没有 JOB / operation 概念**——完全凭 step 区间硬编码状态。

**DockerFactory 现状**（`sim_server.py:459-489` `_build_frame`）：
```json
{ "id": 0, "location": [3, 4], "current_op": [3, 2] }
```
**已经有 op 级信息**了（`current_op = (job_id, op_id)`），但缺：op 在 JOB 中是第几道 / 共几道、op 已执行步数 / 预计步数、machine 队列长度、machine 自身状态（IDLE/WORKING/BROKEN）。

调度场景下用户真正关心的：
- 当前 machine 正在处理哪个 JOB 的哪道 op？
- 这道 op 占整个 JOB 的进度（第几道 / 共几道）？
- 这道 op 本身的执行进度（已加工步数 / 预计步数）？
- machine 队列里还排着多少 op？可以展示一下machine当前积压的op。

### 1.2 需求拆解

**R1.1 — 单 machine 双进度条**
machine本身3D资产旁边可以放一个machine card，显示当前OP与machine积压的op的相关信息。
每个 machine 卡片增加一条"当前 op 进度条"，与现有 machine 整体进度并列：
- 上条：当前 op 步数进度（0–100%，按已加工步数 / 该 op 预计 proc_time 计）
- 下条：op 在 JOB 中的位置（`index / of`，按工序步数计数，**不做时间估算**）

> 评审结论：op 进度按工序步骤计数（用户在原文档已确认），不用时间估算。

**R1.2 — 状态详情面板**
点击 machine 展开 / 浮层展示：
- `current_job_id` / `current_op_id` / `op_index_in_job` / `op_total_in_job`
- `op_step_done` / `op_proc_time`（已加工步数 / 该 op 预计总步数）
- `queue_length`（machine 待处理 op 数）
- `machine_status`（IDLE / WORKING / BLOCKED / BROKEN）

**R1.3 — Frame schema 升级（两个工厂分别处理）**

DockerFactory（`sim_server.py:_build_frame`）扩展为：
```json
{
  "id": 0,
  "location": [3, 4],
  "status": "WORKING",
  "current_op": {
    "job_id": 3, "op_id": 2,
    "index_in_job": 2, "total_in_job": 4,
    "step_done": 5, "proc_time": 12
  },
  "queue_length": 2
}
```
（`current_op` 为 `null` 时表示 machine 空闲；老字段 `(job_id, op_id)` 二元组废弃，统一成对象。）

StaticFactory（`StaticFactoryProxy.py:generate_machine_states`）扩展为：
```json
"M1": {
  "id": "M1",
  "status": "WORKING",
  "load": 60,
  "current_op": { "job_id": "J1", "op_id": "J1-O2", "index_in_job": 2, "total_in_job": 4, "step_done": 5, "proc_time": 12 },
  "queue_length": 2
}
```
（StaticFactory 数据是造的，编一套与轨迹 55 步自洽的 JOB/op 演进脚本即可，反正是前端开发兜底用。）

**Schema 差异声明**（前端必须兼容）：
- StaticFactory：`machines` 是 **dict**，key 为字符串 `"M1"`
- DockerFactory：`machines` 是 **list**，元素 `id` 为 int

前端 `components/FactoryVisualization/*` 需要做一层归一化（提供 `iterMachines(snapshot)` 工具），后续两个工厂共用一套渲染逻辑。

**R1.4 — 向后兼容**
前端发现 `current_op` 是 null / 缺字段时，降级到只显示 machine 状态，不画 op 进度条。避免后端字段没升级完时前端崩。

### 1.3 待澄清

- [x] AGV 是否细化：本期不做（用户已确认 pending，只做 MACHINE 和 JOB）
- [ ] 进度条颜色规范（op 进行中 / 阻塞 / 即将超时）：自行设计，合理即可

## 2. 模块二：日志导出与分析

### 2.1 核心理念

**绘图模板与指标解耦**：
- 指标 = 一组可命名的数值序列（throughput、utilization、makespan、AGV_idle_rate …）
- 图模板 = 一组绘图的"形状"（折线、柱状、堆叠面积、甘特、热力 …）
- 用户在前端：选若干指标 × 选一种图模板 → 出图

不要把"throughput 折线图"硬编码，而是"指标[throughput] + 模板[line]"的组合。

### 2.2 需求拆解

**R2.1 — 日志格式规范**
- 定义天工日志 schema（事件类型、时间戳、字段约束）
- 覆盖：machine 状态变更、AGV 传输、JOB 完成/开始、调度决策、异常
- 输出格式建议：JSON Lines（每行一事件），便于流式解析
- 文档化到 `docs/`，executor 侧统一通过 `sky_logs/logger.py` 写入

**R2.2 — 日志导出接口（后端）**
- `POST /api/logs/export` — 按时间窗 / 实例 / 事件类型筛选，打包下载（.jsonl + 可选 .csv）
- 支持按 `run_id` 导出"某次运行"的全部日志

**R2.3 — 日志上传接口（后端）**
- `POST /api/logs/upload` — 接受 `.jsonl` / `.zip`，校验 schema，落地到文件目录（不必上 DB）
- 返回 `run_id` 供后续分析引用

**R2.4 — 离线日志分析（前端 + 后端）**
- 后端：`POST /api/logs/{run_id}/analyze` — 解析日志，计算指标字典
  ```json
  { "metrics": {
      "throughput": [/* 时间序列 */],
      "machine_util": {"M1": [...], "M2": [...]},
      ...
  } }
  ```
- 前端：
  - 指标选择器（multi-select）
  - 图模板选择器（line / bar / stacked-area / gantt / heatmap）
  - 轴绑定（哪个指标做 X、哪个做 Y、哪个做 groupBy）
  - 预设组合（"常用：throughput 折线 + utilization 柱状"）

**R2.5 — 自定义模板**
- 用户可保存"指标 + 模板 + 轴绑定"为命名 preset，localStorage 持久化
- 后期可演进为服务端 preset 共享

**与两个工厂的关系**：本模块是独立的离线分析能力，日志来源是 DockerFactory（StaticFactory 是 mock，没有分析价值）。日志写入由 `sim_server.py` 配合 `sky_logs/logger.py` 产出；分析接口由 skyengine 后端新增 `application/backend/logs/` 提供，不绑定到任何具体工厂 proxy。

## 3. 模块三：批处理模块

### 3.1 目标

把 `finalpro/SkyEngine/docker-compose.yaml` 这套实验编排（engine on :8080 + mapf + fjsp）封装成"天工前端一键触发"的功能。这个也请你走dockerproxy吧，前端新加一个组件触发，dockerproxy中新添加一个启动docker compose文件的脚本。当前项目已有对应的实现，模仿着写下就行，问题不大。

**注意**：这其实就是 DockerFactory 已经在用的那套容器编排（`DockerProxy.py` 通过 docker compose 拉起 engine/mapf/fjsp），区别在于：
- DockerFactory 是**在线交互式仿真**：用户在前端选算法 → 看实时动画
- 批处理是**离线批量实验**：用户提交多组参数 → 后台跑完 → 回传指标 / 日志 / 产物

两者共用 engine 容器和 `sim_server.py`，但执行节奏和结果回传完全不同。

### 3.2 需求拆解

**R3.1 — 批处理表单（前端）**
参数对齐 `docker-compose.yaml` 里的环境变量：
- `FJSP_INSTANCE`、`MAPF_INSTANCE`
- `NUM_MACHINES`、`NUM_AGV`、`NUM_SIZE`、`NUM_DENSE`、`NUM_RADIUS`、`NUM_STEPS`
- `SEED`、`CUDA_VISIBLE_DEVICES`
- 可选：自定义 image tag（`SKYENGINE_IMAGE` / `FJSP_IMAGE` / `MAPF_IMAGE`，与 `DockerProxy.ALGORITHM_MAP` 对齐）

支持批量：一次提交多组参数（网格扫描 / 列表），后端串行或并发跑。

**R3.2 — 后端批处理服务**
- 新增 `application/backend/batch/` 模块
- `POST /api/batch/submit` — 接收任务参数（单条或多条），返回 `batch_id`
- `GET /api/batch/{batch_id}/status` — 查询进度（pending / running / done / failed）
- `GET /api/batch/{batch_id}/results` — 拉取结果列表
- SSE：`/stream/batch/{batch_id}` 推送实时进度

**R3.3 — 容器拉起方式（两种方案，待评审）**

| 方案                        | 描述                                  | 优点   | 缺点                     |
| --------------------------- | ------------------------------------- | ------ | ------------------------ |
| A. 直接 `docker compose up` | 后端 shell 调 compose，每任务一组容器 | 隔离好 | 启动慢、资源占用高       |
| B. 调用常驻 engine 服务     | engine 常起，天工后端 HTTP 调 :8080   | 快     | 并发管控、参数注入要约定 |

> **与 DockerFactory 的关系**：DockerFactory 已经用了**类方案 A**（按需 `docker compose up engine` → `up mapf fjsp` → `/sim/create` `/sim/play`，stop 时 `compose down`）。批处理建议沿用同一套 `DockerProxy` 抽象，复用 `_compose()` 辅助方法，只是把"实时 SSE 透传"换成"跑完收日志"。
>
> **建议**：开发期用 B（engine 常驻、快速迭代），生产 / 多用户用 A。先实现 B。

**R3.4 — 结果产物**
- 每个任务一份产物清单：日志文件路径、指标 JSON、（可选）SVG / 检查点
- 产物路径指向 finalpro 的 `sky_logs/`（已 bind mount 出来）
- 与模块二打通：批处理结果可直接跳转到"日志分析"页用 `run_id` 分析

## 4. 横向问题

- **权限 / 多用户**：本期单机单用户演示，先不做鉴权，但日志 / 批处理路径预留 user 维度字段
- **数据流一致性**：模块一扩展 frame 字段、模块二定义日志格式、模块三产物含日志 → 三者最好同时定 schema，避免三套不一致
- **国际化**：UI 文案先中文，沿用现有风格
- **与 DockerFactory 的耦合点**：批处理复用 `sim_server.py` + `DockerProxy.ALGORITHM_MAP` + docker-compose-online.yaml，与 DockerFactory 是"同一套引擎的两种用法"，不要各搞一套编排
