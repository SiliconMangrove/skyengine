## 0. 目标与边界

### 0.1 目标
- Episode 跑完后，把这一段会话的 **frames / metricsTimeline / events** 完整快照下来
- 在事件 tab 内能看到所有历史样本，点进去能用同一套 Hook 看图
- 能把样本下载下来（走服务端 `Content-Disposition`，模拟真实生产链路）
- StaticFactory 作为测试床，**不引入后端存储**，纯内存驻留即可

### 0.2 非目标
- 不做后端持久化（重启即失，足够演示）
- 不做权限/审计/分享
- 不做跨 episode 对比（先做单 episode 详情）

### 0.3 关键决策

| 维度 | 选型 | 理由 |
| --- | --- | --- |
| 存储层 | 前端 Pinia store + 内存 | 演示优先，避免后端复杂度 |
| 下载链路 | 走后端 `POST /analysis/export` 模拟 | iframe / CSP 场景下纯前端 Blob 易被拦，服务端 `Content-Disposition` 更可靠 |
| Hook 复用 | 同一份 Hook 函数喂两种 ctx | 实时数据（live）和存档数据（archive）共用算法 |
| 入口位置 | 事件 tab 内嵌 + 头部跳列表按钮 | 事件 tab 天然是"日志"区，无需新增独立导航 |

### 2.2 Run ID 生成规则

```js
function genRunId() {
  runIdSeq += 1
  const ts = new Date()
  return `run_${stamp}_${String(runIdSeq).padStart(3, '0')}`
  // 例如：run_20260622_143501_001
}
```

- 时间戳到秒，肉眼可读
- 末尾 3 位序号防同秒多次 finalize 冲突

### 2.3 Summary 派生逻辑

```js
function computeSummary(frames, metricsTimeline, events) {
  const lastFrame = frames[frames.length - 1] || {}
  const jobs = lastFrame.jobs || []
  const completedJobs = jobs.filter(j => j.is_completed).length

  const utils = metricsTimeline
    .map(p => p.metrics?.machine_utilization)
    .filter(v => typeof v === 'number')
  const avgUtilization = utils.length > 0
    ? utils.reduce((a, b) => a + b, 0) / utils.length
    : null

  const eventCounts = events.reduce((acc, e) => {
    const t = e?.type || 'unknown'
    acc[t] = (acc[t] || 0) + 1
    return acc
  }, {})

  return { total_steps: frames.length, completed_jobs, total_jobs: jobs.length,
           avg_utilization: avgUtilization, event_total: events.length, event_counts: eventCounts }
}
```

## 三、Hook 复用机制（澄清"离线 Hook"命名）

### 3.1 命名说明
**没有"离线 Hook"这个东西**。早期注释里把存档数据叫 "offline"，是误导。本质是：

> Hook 是一段纯函数 `(ctx) => series`，对数据来源零感知。同一份 Hook 代码可以喂两种 ctx：实时（live）和存档（archive）。

后续注释里统一用 `source: 'live' | 'archive'` 替代原 `online/offline` 字样。

### 3.2 Hook 契约

```js
// src/hooks/metrics/<id>.js
export default {
  id: 'machine_util',
  label: '机器利用率',
  unit: '%',
  multiSeries: false,                    // 可选，标识是否多 series
  series: (ctx) => [{ x, y }, ...]       // 或多 series: [{ name, data, color }]
}
```

`ctx` 由 caller 归一化：

| 字段 | live 来源 | archive 来源 |
| --- | --- | --- |
| `metrics` | `monitorStore.metricsTimeline` | `run.metricsTimeline` |
| `frames` | `factoryStore.historyBuffer` | `run.frames` |
| `events` | `monitorStore.events` | `run.events` |
| `source` | `'live'` | `'archive'` |

### 3.3 注册中心

```js
// src/hooks/metrics/index.js
import machineUtil from './machine_util.js'
import agvActivity from './agv_activity.js'
import jobProgress from './job_progress.js'
import eventTimeline from './event_timeline.js'

export const HOOKS = { machine_util, agv_activity, job_progress, event_timeline }
export const listHooks = () => Object.values(HOOKS)
```

### 3.4 当前内置 Hook

| ID | Label | 数据源 | multiSeries |
| --- | --- | --- | --- |
| `machine_util` | 机器利用率 | `metrics[*].machine_utilization` | 否 |
| `agv_activity` | AGV 活动数 | `frames[*].grid_state.is_active`（true 计数） | 是 |
| `job_progress` | Job 完成进度 | `frames[*].jobs`（`is_completed` 累计） | 是 |
| `event_timeline` | 事件累计 | `events[*]` 按 `type` 分组累计 | 是 |

### 4.2 finalizeFromStores 关键点

```js
function finalizeFromStores(factoryStore, monitorStore, meta = {}) {
  // ⚠️ 深拷贝必须，否则 store.reset() 会把 run 数据一起清空
  const frames = JSON.parse(JSON.stringify(factoryStore.historyBuffer || []))
  const metricsTimeline = JSON.parse(JSON.stringify(monitorStore.metricsTimeline || []))
  // monitor store 暴露的是 events 别名（不是 eventQueue），双名兜底
  const events = JSON.parse(JSON.stringify(monitorStore.eventQueue || monitorStore.events || []))

  const summary = computeSummary(frames, metricsTimeline, events)
  const run = { id: genRunId(), frames, metricsTimeline, events, summary, ...meta }
  runs.value.unshift(run)   // 新的放最前
  return run
}
```

**踩过的坑**：原本写的是 `monitorStore.eventQueue`，但 monitor store 暴露的字段名是 `events`（内部 ref 别名）。结果 events 永远是空数组。修正为双名兜底。

### 4.3 Episode 结束自动调用

`composables/useSimulationConfig.js` 的三个执行路径（executeFrontend / executeBackend / executeDocker）在 `onFinish` 之前统一调用：

## 五、服务端模拟下载

### 5.1 为什么需要后端

纯前端 `Blob + a.click()` 在以下场景会失败：
- iframe / 嵌入式 webview
- 严格 CSP 策略
- 跨域 window 上下文

生产环境一定是后端返回带 `Content-Disposition: attachment` 头的响应，浏览器原生触发下载，可靠性最高。

### 5.2 StaticFactory 后端 mock 路由

```python
# application/backend/core/StaticFactoryProxy.py
def _register_analysis_routes(self):
    @RouteRegistry.register_route("/analysis/export", method="POST")
    async def api_analysis_export(payload: dict):
        run_id = (payload or {}).get("id") or f"run_{int(time.time())}"
        body = json.dumps(payload, ensure_ascii=False, indent=2).encode("utf-8")
        safe_id = "".join(c if c.isalnum() or c in "-_" else "_" for c in str(run_id))
        return Response(
            content=body,
            media_type="application/json",
            headers={"Content-Disposition": f'attachment; filename="{safe_id}.json"'},
        )
```

**设计要点**：
- 前端把内存中的 run JSON POST 过来
- 后端加 `Content-Disposition: attachment` 头原样回吐
- 浏览器看到 attachment 头会自动触发下载
- **没有真正落盘**，纯 mock —— 未来 PacketFactory 接真实落盘时，把这里换成 `FileResponse(zip_path)` 即可，调用方零改动

### 5.3 server.py 注册

切换到 static_factory / northeast_center 时也要调 `RouteRegistry.register_to_app(app)`（原本只 packet_factory 走）：

```python
elif factory_type in ("static_factory", "northeast_center"):
    await current_factory_proxy.initialize()

```

### 5.4 前端调用与 fallback

```js
// analysisLog.js
async function _downloadViaBackend(payload) {
  const url = `${API_BASE}${API_ROUTES.ANALYSIS_EXPORT}`
  try {
    const resp = await fetch(url, { method: 'POST', body: JSON.stringify(payload), ... })
    if (!resp.ok) throw new Error(`HTTP ${resp.status}`)
    const blob = await resp.blob()
    // 从 Content-Disposition 解析文件名，否则用 payload.id
    const cd = resp.headers.get('content-disposition') || ''
    let filename = `${payload.id || 'run'}.json`
    const m = /filename="?([^";]+)"?/.exec(cd)
    if (m) filename = m[1]
    _triggerBrowserDownload(blob, filename)
    return true
  } catch (e) {
    // 后端不可用（开发态没起服务）→ 回退纯前端 Blob
    console.warn('[analysisLog] backend export failed, fallback:', e)
    const blob = new Blob([JSON.stringify(payload, null, 2)], { type: 'application/json' })
    _triggerBrowserDownload(blob, `${payload.id || 'run'}.json`)
    return false
  }
}
```

**双保险**：后端在 → 走后端；后端没起 → 自动降级到前端 Blob，不会硬失败。

### 7.2 返回工厂的关键修复

**踩过的坑**：FactoryView 有 `onBeforeRouteLeave → cleanupFactory()` 会断后端 + 清 store；组件卸载时 `onBeforeUnmount` 也调 `factoryStore.clearAll()`。所以无论怎么"返回工厂"，都得重走 loading + switch 流程，视觉上像"回到选择器再选一次"。

**三层修法**：

| 层 | 措施 | 效果 |
| --- | --- | --- |
| `onBeforeRouteLeave(to)` | `to.path.startsWith('/analysis')` 时只写 session 标记、不调 cleanup | 去往 /analysis 时后端连接 + historyBuffer 全保留 |
| `onBeforeUnmount` | 检测到 session 标记存在则跳过 `clearAll` | 组件卸载也不丢数据 |
| setup 阶段初始化 ref | 同步读 `sessionStorage[FACTORY_SESSION_KEY]`，直接初始化 `isInFactory` / `currentFactoryId` | 首帧就用对的 state 渲染，**完全看不到选择器闪现** |

```js
// 关键：必须在 setup 同步执行，不能放 onMounted
const _initial = _readInitialFactoryState()   // 读 sessionStorage + route.query
const _initialValid = _initial.factoryId &&
  factoryStore.getFactories().some(f => f.id === _initial.factoryId)

const isInFactory = ref(!!(_initialValid && _initial.fast))
const currentFactoryId = ref(_initialValid ? _initial.factoryId : null)
```

**为什么不能放 onMounted**：`onMounted` 在首次渲染**之后**触发，初始值 `isInFactory=false` 会先画出选择器，onMounted 再改成 true，用户就看到"选择器闪一下再切工厂"的 bug。

## 九、设计纪律

1. **零后端存储**：演示优先，重启即失；未来接 PacketFactory 时把 mock download 换成 `FileResponse` 即可
2. **Hook 复用单一权威**：实时和存档共用一份 Hook 代码，差异只在 ctx 归一化层
3. **降级而非崩溃**：
   - 后端没起 → 自动降级到前端 Blob 下载
   - 单个 Hook 报错 → `safeSeries` 兜底返回空，不影响整页
   - Run 不存在 → 详情页显示"未找到样本"空状态
4. **样式复用**：LineChart 是泛化组件，被所有 Hook 共用；EventPanel 内嵌列表样式独立但视觉风格统一
5. **sessionStorage 而非 localStorage**：返回工厂的"活动会话"是 tab 级状态，关闭 tab 即失效；不污染长期存储
