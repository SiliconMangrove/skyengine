## 0. 已拍板的决策与总原则

### 0.1 两个对齐方向（已定）
| 维度 | 决策 | 谁动 |
|---|---|---|
| events 通道 | 独立流（不嵌在 frame 里） | StaticFactory 拆出独立 events 流 |
| machines 结构 | `list` + `id` 字段 | StaticFactory 由 dict 改 list |

### 0.2 总原则：StaticFactory 全面向容器化 SkyEngine 对齐
**当两者不一致，一律以容器化 SkyEngine 的输出为 canonical，由 StaticFactory 改自己。** 容器化 engine 是真实引擎，它的形状有结构性理由（见 `0627帧与事件相关设计.md`）；StaticFactory 是前端剧本，改它没有信息损失。

### 0.3 架构取舍（已定）
**不引入 LogCollector 组件。** 归一化在**各 proxy 产出时就近完成**（每个 proxy 是自己的收口），落盘挂 `server.py` 已有的 play/stop 生命周期、复用已有 history 模块。不新增常驻组件、不新增 `/logs` 路由。

## 一、现状对照（决定 StaticFactory 要改什么）

| 维度 | 容器化 SkyEngine（canonical） | StaticFactory（要改） |
|---|---|---|
| proxy 角色 | passthrough（透传 engine） | producer（自己跑 `_run_mock`） |
| SSE 模式 | push（`*_stream`） | pull（`get_*_events`） |
| events 通道 | 独立 `/stream/events` ✓ | **无**（嵌在 snapshot.events） |
| events schema | `{timestamp, type, message, level}` | `{step, title, message, type}` |
| events.type 语义 | 业务名（`machine_start_op`…） | 级别（`info/success/task/error`） |
| machines | `[{id:int, location, status, current_op, queue_length}]` | `{"M1": {...}}` |
| metrics | `{status, step, metrics:{...}, metrics_reward}` | `{step, machine:{data,labels}, agv, job, keyMetrics}` |
| frame 顶层字段 | `timestamp/env_timeline/grid_state/machines/jobs/active_transfers` | 一致 ✓（除多了 `events` 字段） |

**容器化侧不改**，它就是 canonical 的源头。

## 二、Canonical Schema（= 容器化 SkyEngine 的输出）

### 2.1 Frame（每步一条，state 流）
```jsonc
{
  "timestamp": "T+{step}s",
  "env_timeline": "<string>",
  "grid_state": { "positions_xy": [[x,y],...], "is_active": [bool,...] },
  "machines": [
    { "id": 1, "location": [x,y], "status": "WORKING"|"IDLE",
      "current_op": { "job_id":.., "op_id":.., "step_done":.., "proc_time":.. } | null,
      "queue_length": 0 }
  ],
  "jobs": [ { "job_id":.., "is_completed": bool, "progress": {done,total}, "ops":[...] } ],
  "active_transfers": [ { "task_id":.., "job_id":.., "op_id":.., "source":[..], "destination":[..]|null } ]
}
```
**frame 里没有 `events` 字段。** events 走独立流。

### 2.2 Metrics（每步一条，metrics 流）
```jsonc
{ "status": "running", "step": 42, "metrics": { "machine_utilization": 0.42, ... }, "metrics_reward": 0.0 }
```

### 2.3 Event（仅跃迁时，独立 events 流）
```jsonc
{ "timestamp": "T+42s", "type": "machine_start_op", "message": "M3 开始 Job2-Op1", "level": "info" }
```
- `type`：业务事件名（`sim_started / machine_start_op / machine_idle / transfer_started / transfer_completed`），引擎相关词汇表
- `level`：`info / success`（未来可扩 `warning / error`）

### 2.4 Run（episode 聚合，落盘单位）
对齐 history 模块的 `RunData`（`application/backend/history/models.py:26`）：
```jsonc
{
  "schema_version": "1.0",
  "id": "run_...", "factory_id": "...", "algorithm": "...",
  "created_at": "ISO", "started_at": "T+0s", "finished_at": "T+54s", "total_steps": 55,
  "frames":           [ /* 2.1 Frame[] */ ],
  "metrics_timeline": [ /* 2.2 Metrics[] */ ],
  "events":           [ /* 2.3 Event[] */ ],
  "summary": { "total_steps":.., "completed_jobs":.., "total_jobs":..,
               "avg_utilization":.., "event_total":.., "event_counts": {...} }
}
```

## 三、StaticFactory 对齐清单（在 StaticFactoryProxy 内就近改）

所有改动集中在 `StaticFactoryProxy.py` + `FactorySimulator`，不动容器化 engine。

### 3.1 events 拆独立流（最关键）
**现状**：`get_events(step)` 返回 `{step,title,message,type}`，嵌在 snapshot.events；`/stream/events` 对 StaticFactory 返回 "No event source"（`server.py:221`）。

**改法**：
1. `FactorySimulator.get_events(step)` 改返回 canonical 形状：`{timestamp:"T+{step}s", type, message, level}`
   - 把硬编码事件的 `type`（info/success/task/error）映射为 `level`（info/success）
   - `type` 字段尽量套用容器化词汇表：AGV 取货→`transfer_started`、AGV 卸货→`transfer_completed`、机器开始加工→`machine_start_op`、机器空闲→`machine_idle`；实在套不上的叙事性事件用 `type:"narrative"` 兜底
   - `title` 字段丢弃（canonical 没有），内容并进 `message`
2. snapshot 里的 `events` 字段**移除**（frame 不再带 events）
3. StaticFactoryProxy **加 events 通道**：要么实现 `events_stream()`（push 模式，与 DockerProxy 同构，最干净），要么加 `get_events_events()` 走 pull。**推荐 push 模式**——这样 server.py 的 `hasattr(proxy,"events_stream")` 分支能直接覆盖三条通路，统一处理。

### 3.2 machines dict → list
**现状**：`generate_machine_states` 返回 `{"M1": {...}}`（`StaticFactoryProxy.py:529`）。

**改法**：改成返回 list，元素结构对齐 canonical：
```jsonc
[{ "id": "M1", "location": [x,y], "status": "WORKING"|"IDLE",
   "current_op": {...}|null, "queue_length": 0 }]
```
- `id` 保留字符串 `"M1"`（StaticFactory 原生就是这名，不强转 int——id 是引擎相关标识，canonical 只约束"必须是 list + 有 id 字段"，不强约束类型）
- 加 `location` / `status` / `current_op` / `queue_length` 字段，剧本里有就填、没有填 null/默认

**前端影响**：所有 `machines["M1"]` 的消费点改成 `machines.find(m => m.id === "M1")`，或前端自己 `reduce` 成 map。前端 hooks 的 `_normalize.js` 已在处理类似差异，可统一收口。

### 3.3 metrics 形状对齐
**现状**：`{step, machine:{data,labels}, agv:{data}, job:{data}, keyMetrics:{...}}` 是给柱状图预聚合的展示型结构。

**改法**：`generate_metrics_data(step)` 改返回 canonical：
```jsonc
{ "status": "running", "step": N, "metrics": { "machine_utilization": 0.xx, ... }, "metrics_reward": 0.0 }
```
- 剧本是 mock，`metrics` 子字典里填 `machine_utilization`（必填，Hook 要用）+ 其他可枚举指标
- 旧的 `machine/agv/job/keyMetrics` 展示结构**废弃**；消费它的柱状图组件改读 `metrics` 字典（或读 frame 的 machines/jobs 派生）

**前端影响**：DashboardPanel / BarChart 若直接吃旧 `machine.data` 数组，需改为从 canonical metrics 或 frame 派生。这是本方案前端侧最大改动点，需单独盘点。

### 3.4 三条流就位后
StaticFactory 也拥有 `state_stream / metrics_stream / events_stream`（push 模式），server.py:118/167/216 的 `hasattr` 分支天然覆盖，**server.py 不用改 SSE 路由逻辑**。

## 四、落盘：复用 history 模块 + server.py 生命周期（无 LogCollector）

### 4.1 现状断点
- `server.py:541` play 时 `create_run` ✓（建 SQLite 元数据行）
- `server.py:391/652` stop 时 `complete_run(current_run_id)` ✗ ——**没传 `run_data`**，JSON 详情文件从不生成
- `append_run_data` ✗（零调用）

### 4.2 改法（server.py 内就近，~20 行）
1. 模块级加一个 episode buffer：
   ```python
   current_run_buffer = {"frames": [], "metrics_timeline": [], "events": []}
   ```
2. `create_run` 时清空 buffer
3. 三个 SSE 端点的 `generate()` 里，每 yield 一条数据前 append 到 buffer 对应列表（pull 和 push 两个分支都加）
4. `complete_run` 改成：
   ```python
   run_data = RunData(meta={...}, metrics_timeline=current_run_buffer["metrics_timeline"],
                      events=current_run_buffer["events"], key_snapshots=current_run_buffer["frames"])
   history_manager.complete_run(current_run_id, metrics_summary=..., run_data=run_data)
   ```
5. `summary` 由后端在 complete 时计算（job 完成数、平均利用率、event_counts 按 type 统计）

**没有新组件**，就是 server.py 现有生命周期里挂一道累积 + flush。

### 4.3 不走流式落盘
默认 episode 结束一次性 flush（简单、IO 少）。`append_run_data` 暂不用，留作超长 episode 的分段 flush 备选。

### 4.4 前端读历史
`utils/api.js:43-48` 已定义 `HISTORY_*` 路由常量（当前是死代码），直接用：
- 列表 → `GET /history/runs`
- 详情 → `GET /history/run/{id}`（返回完整 RunData）
- 删除 → `DELETE /history/run/{id}`
- 导出 → 需给 history routes 补一个 `GET /history/run/{id}/export`（返回 `Content-Disposition: attachment`）

## 五、前端改动清单

### 5.1 `analysisLog` store 重构
- `runs` 只存 `RunMeta`（id/factory_id/algorithm/created_at/summary），**不再持 frames**
- 异步 action 改走 `/history/*`：
  ```js
  async fetchList()                 // GET /history/runs
  async fetchRun(id)                // GET /history/run/{id}
  async deleteRun(id)               // DELETE /history/run/{id}
  ```
- episode 结束**不再前端上传**（后端 server.py 已 flush），前端只需 `fetchList()` 刷新

### 5.2 `AnalysisOverlayPanel.vue`
- 列表来自 `fetchList()`，详情来自 `fetchRun(id)`，Hook 图表复用不变
- 上传日志按钮（⬆，已预留）：本地 JSON → 校验 `schema_version` → `POST /history/run`（history routes 需补 POST 端点）→ 刷新。**待 canonical 最终确认后实现**

### 5.3 实时事件流
- EventPanel 实时事件改订阅 `/stream/events`（StaticFactory 对齐后也有这条流了）
- 不再从 `factoryStore.historyBuffer` 里翻 events

### 5.4 `_normalize.js` 处置
- canonical 统一后，前端散点归一逻辑理论上可删
- 建议**先保留**作 defense-in-depth，等两条通路都验证稳定后再清

### 5.5 machines 消费点
- 所有 `machines["M1"]` → `machines.find(m => m.id === "M1")` 或前端 reduce 成 map

## 八、与旧方案（LogCollector 版）的差异

| 维度 | 旧版（LogCollector） | 本版（proxy 就近对齐） |
|---|---|---|
| 归一化位置 | 独立 LogCollector 组件 | 各 proxy 产出时就近 |
| 新增组件 | LogCollector（常驻） | 无 |
| 新增路由 | `/logs/*` | 无（复用 `/history/*`） |
| canonical 定义 | 中性契约（category 统一 + type 引擎相关） | 直接等于容器化 SkyEngine 输出 |
| category 字段 | 有（统一分类层） | 无（用容器化的 type 词汇表，StaticFactory 套用） |
| 复杂度 | 高（新组件 + 新路由 + 双范式映射） | 低（改一个 proxy + server.py 加 buffer） |
| StaticFactory 角色 | 与容器化平级，各自归一 | 从属于容器化，全面对齐 |
