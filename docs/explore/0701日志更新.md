## 术语表（统一口径，全文沿用）

| 术语 | 实体 | 一句话 |
|---|---|---|
| **日志模块** | `analysisLog` store（`stores/analysisLog.js`） | 整个离线分析子系统的本体 |
| **Run** | `analysisLog.runs[]` 的一条记录 / `dataset/run/` 下的一个 JSON 文件 | **一次完整仿真跑完的存档**——一盒录像带 |
| **summary** | `run.summary` 字段 | Run 的统计标签（总步数/完成率/平均利用率/事件数） |
| **三流** | `run.frames[]` / `run.metricsTimeline[]` / `run.events[]` | Run 内部的三条数据轨道（画面/指标/事件） |
| **frame** | `run.frames[]` 一个元素 | 一步的工厂状态快照（视频轨的一帧） |
| **metrics** | `run.metricsTimeline[]` 一个元素 | 一步的指标数值（数据轨的一帧） |
| **event** | `run.events[]` 一个元素 | 一步发生的语义事件（弹幕轨的一条） |
| **archive** | `dataset/run/` 目录 | Run 的后端持久化仓库（0701 新增方向） |
| **finalize** | `finalizeFromStores()` | episode 结束时把三流深拷贝封存成 Run 的动作 |

**类比**：日志模块 = 录像带柜子；Run = 一盒录像带；summary = 录像带上的比分标签；三流 = 录像带里的视频/数据/弹幕三条轨道；archive = 柜子的实物储藏室。

## 0. 已落地的关键决策（与 0627 的差异）

| 维度 | 0627 设想 | 0701 实际落地 |
|---|---|---|
| StaticFactory 数据源 | 后端 proxy 产出 → SSE 流 | **前端剧本 `fullSystemTest.js` 直推 store**，后端 proxy 降为占位 |
| StaticFactory 三条流切 push | 整体切 async generator | **不做**——后端 proxy 是占位，无流可切 |
| 后端 history 落盘 | server.py episode buffer + complete_run run_data | **0701 修订**：episode 结束后端落盘到 `dataset/run/<run_name>.json`（详见 §三·3.5 与 `0701日志详细设计.md`） |
| canonical schema 边界 | 容器化为基线 + StaticFactory 扩字段 | **严格对齐容器化**（metrics 字段以容器化为权威） |
| 事件 category 字段 | 引入统一分类层 | **前端暂不引入**（容器化侧升级后再反向推动） |
| 上传日志分析 | 后端 `POST /history/run` | **0701 修订**：上传走前端 file input（用户任选本地路径），导入后默认写盘到 `dataset/run/` |

**核心原则修订（0701）**：日志模块从"纯前端内存"升级为**后端文件系统持久化**——`dataset/run/` 作为 Run 仓库，前端 `analysisLog.runs[]` 退化为该仓库的内存缓存。StaticFactory 与容器化通路共用同一 Run 仓库。详见 `0701日志详细设计.md`。

## 一、Canonical Schema（统一数据契约）

两条通路产出**同形**数据，前端 hook / 面板零分支消费。

### 1.1 Frame（每步一条，state 流）
```jsonc
{
  "timestamp": "T+{step}s",
  "env_timeline": "<string>",
  "grid_state": { "positions_xy": [[x,y],...], "is_active": [bool,...] },
  "machines": [
    { "id": "M1" | 1, "location": [x,y], "status": "WORKING"|"IDLE"|"BROKEN",
      "current_op": { "job_id":.., "op_id":.., "step_done":.., "proc_time":.. } | null,
      "queue_length": 0,
      "load": <可选，StaticFactory 私有> }
  ],
  "jobs": [
    { "job_id":.., "release":.., "due":..|null,
      "is_completed": bool, "completion_time":..,
      "progress": { "done":.., "total":.. },
      "ops": [ { "op_id":.., "status":"PENDING"|"PROCESSING"|"FINISHED", "assigned_machine":..,
                "arrive_machine_at":.., "start_process_at":.., "finish_process_at":..,
                "wait_for_machine_time":.., "step_done":.., "proc_time":.. } ] }
  ],
  "active_transfers": [
    { "task_id":.., "job_id":.., "op_id":.., "source":[..], "destination":[..]|null }
  ]
}
```
- `machines` 强制 **list + id 字段**；id 类型不强约束（StaticFactory 用 string `"M1"`，容器化用 int `1`）
- frame **不带** `events` 字段（events 走独立流）
- `load` 是 StaticFactory 私有扩展字段，容器化无；hook 用 `queue_length` 兜底

### 1.2 Metrics（每步一条，metrics 流）
```jsonc
{
  "status": "running" | "stopped",
  "step": 42,
  "metrics": {
    "machine_utilization": 0.42,            // 必填，前端 machine_util hook 消费
    "machine_non_processing_time_mean": ..,
    "machine_load_variance": ..,
    "operation_queue_waiting_time_mean": ..,
    "efficiency": ..,                        // StaticFactory 私有，容器化可缺省
    "agv_loaded_utilization": ..,
    "agv_busy_utilization": ..,
    "agv_travel_time_total": ..,
    "agv_waiting_time_total": ..,
    "swap_conflict_count": ..,
    "tasked_stationary_count": ..
  },
  "metrics_reward": 0.0
}
```
- 字段集**以容器化 MetricsHub 为权威**（见 0627static对齐难点 §二），StaticFactory 是 mock，可填部分字段，缺失项 hook 自然降级
- 废弃旧展示型 `{machine:{data,labels}, agv, job, keyMetrics}`；monitor.pushMetrics 内部 dual-write 派生它仅供老 MetricsPanel 兼容

### 1.3 Event（仅跃迁时，独立 events 流）

**stream 信封**（engine → 前端，容器化侧升级目标，见 §五）：
```jsonc
{
  "step": 42,
  "timestamp": "T+42s",
  "category": "machine_op" | "transfer" | "job" | "agv" | "lifecycle" | "risk" | "system",
  "type": "machine_start_op",       // 引擎相关业务名
  "level": "info" | "success" | "warning" | "error",
  "title": "M3 开始加工",            // ≤20 字短标题
  "message": "M3 开始 Job2-Op1",    // 详细描述
  "payload": { ... }                // 可选结构化字段
}
```

**前端内部表示**（`monitor.eventQueue[]`，pushEvent 后存储）：
```jsonc
{
  "id": <number>,
  "timestamp": "T+42s" | Date,
  "idx": 42,                        // = step
  "type": "machine_start_op",       // 业务名（用于图标精修映射）
  "level": "info" | "success" | "warning" | "error",
  "message": "..."
}
```
- 当前前端 pushEvent **未保留** `category / title / payload`（简化）—— lossy 映射
- 容器化侧 `_diff_and_emit` 升级落地后，如前端需要 category 聚合，再扩 pushEvent 字段

### 1.4 Run（episode 聚合，前端落盘单位）
```jsonc
{
  "schema_version": "1.0",
  "id": "run_20260701_143501_001",
  "factory_id": "static_factory",
  "algorithm": null,
  "created_at": "ISO",
  "started_at": "T+0s",
  "finished_at": "T+54s",
  "total_steps": 55,
  "frames":           [ /* 1.1 Frame[] */ ],
  "metricsTimeline":  [ /* 1.2 Metrics[] */ ],
  "events":           [ /* 1.3 Event[] (前端内部表示) */ ],
  "summary": {
    "total_steps": 55,
    "completed_jobs": 2, "total_jobs": 3,
    "avg_utilization": 0.42,
    "event_total": 6,
    "event_counts": { "sim_started": 1, "narrative": 4, "episode_completed": 1 }
  }
}
```

## 三、日志模块（analysisLog）定型

### 3.1 设计取舍

| 维度 | 选型 | 理由 |
|---|---|---|
| 存储 | 前端 Pinia store + 内存 | 演示优先；StaticFactory 不引入后端复杂度 |
| 生命周期 | 页面刷新即失 | 足够演示；长期持久化由容器化通路走 history 模块 |
| 写入时机 | episode 结束（onFinish 回调） | `useSimulationConfig.snapshotToAnalysisLog` 三条执行路径统一调用 |
| 数据来源 | deep copy 自 factoryStore + monitorStore | 避免后续 store.reset() 把 run 数据清掉 |
| 导出 | 后端 `/analysis/export` mock | `Content-Disposition: attachment` 比 Blob + a.download 在 iframe/CSP 场景更可靠 |
| 导入 | 前端纯本地解析 | 无需后端参与，闭环最短 |

### 3.2 公开接口（`stores/analysisLog.js`）

```js
// 状态
analysisLog.runs              // Run[]
analysisLog.totalRuns         // number
analysisLog.panelOpen         // 浮层开关
analysisLog.initialRunId      // 打开时定位的 runId

// 浮层
analysisLog.openPanel(runId?)
analysisLog.closePanel()

// 写入（episode 结束自动调）
analysisLog.finalizeFromStores(factoryStore, monitorStore, meta) → Run

// 查询 / 删除
analysisLog.getRunById(id)
analysisLog.deleteRun(id)
analysisLog.clearAll()

// 导出
await analysisLog.exportRun(id)              // 已存档 run → 后端下载
await analysisLog.exportLive(factoryStore, monitorStore, meta)  // 当前会话不入库

// 导入（待启用按钮触发）
analysisLog.importRun(payload)               // 校验 → 补 summary → unshift
```

### 3.3 导出链路（已实现）

```
analysisLog.exportRun(id)
   → 取出 run 对象
   → POST /analysis/export（body = run JSON）
   → 后端（StaticFactoryProxy 占位实现）加 Content-Disposition: attachment 头原样回吐
   → 浏览器原生触发下载
   → fallback：后端不可用时降级为前端 Blob + a.click()
```

后端代码（`StaticFactoryProxy.py:_register_analysis_routes`）只是 mock，未来容器化通路接真实落盘时，把这里换成 `FileResponse(zip_path)` 即可，调用方零改动。

### 3.4 导入链路（待启用）

当前 `AnalysisOverlayPanel.vue` 的"⬆ 上传日志分析"按钮 disabled，理由是"待日志格式确定"。**0701 起 canonical schema 已定稿（本文 §一），闭环条件已满足**，按钮可启用。

实现要点（约 30 行代码）：
1. `analysisLog.importRun(payload)`：校验 `payload.frames / metricsTimeline / events` 三字段存在 → 缺 `summary` 自动 `computeSummary` 补算 → 分配新 `id` → `runs.unshift`
2. `AnalysisOverlayPanel.vue` 启用按钮 + 隐藏 `<input type="file" accept=".json">`
3. 文件读取 → `JSON.parse` → try/catch 包裹（格式错误弹 ElMessage.error）→ `importRun` → 自动进详情视图

接受任何符合 canonical Run shape 的外部 JSON：之前导出的 run、容器化通路导出的 episode、未来手工构造的测试样本。

## 五、容器化 engine 侧：`_diff_and_emit` 升级方向

容器化 SkyEngine（外部仓库 `/data1/home/wuhao/project/finalpro/SkyEngine/sim_server.py`）的事件升级详细方案见 `explore/0701diff_and_emit升级.md`。本文只做**意图总结**，便于前端预期落地形态。

### 5.1 一句话意图

把 `_diff_and_emit` 从「内联两维 if-else」演进为「可扩展的 Rule 注册表」，事件信封对齐 §1.3 stream 信封。

### 5.2 结构变化

- **Rule = 纯函数** `(prev_frame, cur_frame, ctx) -> Iterator[Event]`，无副作用、不直接广播
- **主函数退化为调度器**：列表装配 + 循环调用 + try/except 降级
- **顺序即列表顺序**：machine_op → transfer → op_finish → job_completion → agv_state → deadline_risk
- **`_prev_frame` 滚动放 `finally`**（关键不变量：规则异常不污染基线）

### 5.3 规则清单

| ID | 规则 | 类别 | 状态 |
|---|---|---|---|
| R1 | `machine_op_rule` | 纯跃迁 | 现有（迁移） |
| R2 | `transfer_rule` | 纯跃迁 | 现有（迁移） |
| R3 | `job_completion_rule` | 纯跃迁 | **新增**（job.is_completed False→True） |
| R4 | `op_finish_rule` | 纯跃迁 | **新增**（op.status → FINISHED） |
| R5 | `agv_state_rule` | 纯跃迁 | **新增**（grid_state.is_active 翻转） |
| R6 | `deadline_risk_rule` | **阈值**（唯一例外） | **新增**，配置开关 `enable_deadline_risk`，需去重 |

R6 是**首个非纯跃迁规则**，破坏「events 流是纯语义跃迁」原则。这是有意识的让步，新增任何阈值规则需在文档登记，防止 events 流退化为阈值告警流。

### 5.4 容错

- 单条 Rule 抛异常 → 捕获 + 降级为一条 `system/diff_rule_error` warning 事件 + print 日志
- 字段缺失用 `.get()` 默认值容忍
- 主循环与 `_prev_frame` 滚动**永不中断**

### 5.5 实施分阶段

| Phase | 内容 | 验证 |
|---|---|---|
| 1 | 结构重构（拆 Rule） | events 流与升级前**逐字节一致** |
| 2 | 信封升级（加 step/category/title/payload） | 旧前端行为不变（增量字段） |
| 3 | 新增 R3/R4/R5 纯跃迁规则 | events 总数多于现状 |
| 4 | R6 阈值规则（可选开关） | 同一 job 一个 episode 只发一次 |
| 5 | 健壮性收尾（.get + 异常降级 + finally 滚动） | 注入异常主循环不中断 |

**每个 Phase 独立可验证、独立可回滚**。Phase 1-2 必做，Phase 3 强烈建议，Phase 4-5 视前端反馈。

## 六、新增事件的规范（两套通路各自如何加）

> **后续添加事件按本节规范执行。**

### 6.1 通路 A（StaticFactory 前端剧本）

加事件 = 改 `scenarios/fullSystemTest.js` 的 `EVENTS_LOG`：

```js
const EVENTS_LOG = [
  ...,
  { step: <触发步>, type: '<业务名>', level: 'info|success|warning|error', message: '<文案>' },
]
```

约束：
- `type` 优先套用 §4.2 category 表中已有业务名（`machine_start_op / transfer_started / job_completed / ...`）
- 套不上的叙事事件用 `type: 'narrative'` + 合适 level
- 原 title 字段**并入 message**（canonical 无 title）
- 触发步必须与 JOB_SCRIPT / machine 阶段对齐（剧本一致性）
- 图标：`EventPanel.vue` 的 `eventIconMap` 已按 type+level 复合映射，新增 type 想要专属图标就在那里加一行

### 6.2 通路 B（容器化 engine，升级后）

加事件 = 在 `sim_server.py` 新增一个 Rule 函数 + 注册到规则列表一行：

```python
def <name>_rule(prev, cur, ctx):
    # 1. .get() 容忍字段缺失
    prev_X = { ... for x in prev.get("<field>", []) }
    for x in cur.get("<field>", []):
        # 2. 跃迁判定（纯状态机/集合差）
        if <transition>:
            # 3. yield 半成品事件（信封由 _emit_envelope 补齐）
            yield {
                "category": "<cat>",
                "type": "<业务名>",
                "level": "<info|success|warning|error>",
                "title": "<≤20字>",
                "message": "<详细>",
                "payload": { ... },  # 可选
            }

# 注册（列表装配，顺序即求值顺序）
self._diff_rules = [..., <name>_rule]
```

约束（来自 0701diff_and_emit升级.md §五）：
- **纯跃迁优先**：能写成 prev/cur diff 的，绝不引入阈值/计数/时序
- **阈值规则单独标注**：若必须引入阈值（如 deadline_risk），需配置开关 + 去重机制 + 在本文 §4.2 category 表登记
- **绝不直接广播**：Rule 只 yield，`_emit_envelope` 统一发
- **字段缺失容忍**：用 `.get()`，不假设 frame 完整
- **绝不污染 `_prev_frame` 滚动**：异常会被调度器捕获，基线滚动放 finally
- **payload 字段不强统一**：按 category+type 解析，允许 Rule 特化
- 新增 type 同步登记到本文 §4.2 category 表 + StaticFactory 侧 `eventIconMap`

### 6.3 通路 A↔B type 对齐原则

- 通路 B 新增的纯跃迁 type（如 R3 `job_completed`、R5 `agv_state`），通路 A **不强制**同步——StaticFactory 是确定性剧本，事件固定 6 条够用
- 反向：若通路 A 出现某 type 频繁（如 `narrative`），通路 B 不应跟（容器化不应有叙事事件）
- 共享的是 **shape**，不是 type 集合
