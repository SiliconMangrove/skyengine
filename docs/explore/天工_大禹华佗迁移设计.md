## 0. 一句话总览

把大禹/华佗的"**数据 / 模板 / 配置三层分离**"搬到天工：
- **数据层**：用 Hook 把 sim_server 的原始流（metrics / state / events / 日志）归一成"指标序列"
- **模板层**：把现在 `MetricsPanel.vue` 里写死的 line / bar / heatmap 抽成无数据源绑定的通用 Chart 组件
- **配置层**：用 JSON 看板配置文件自由组合「Hook × 模板 × 轴绑定」，在线（DockerFactory 实时仿真）和离线（日志导入）共用同一套渲染

**核心约束（沿用师兄的三条铁律）**：
1. 数据/视图解耦：严禁数据逻辑和图表渲染代码强耦合
2. Hook 扩展：新增指标只写 Hook，不动渲染代码
3. 模板复用：同一套折线模板既能画 CPU 也能画利用率

## 2. 最终形态：在线自定义绘图（天工版大禹）

### 2.1 三层架构落地

```
┌─────────────────────────────────────────────────────────────┐
│ 配置层 dashboard.config.json                                │
│  [{hook:"machine_util", template:"line", xAxis:"step", ...}]│
└────────────────────────┬────────────────────────────────────┘
                         │ 解析渲染
┌────────────────────────▼────────────────────────────────────┐
│ 模板层 components/charts/ (无数据源绑定)                     │
│  LineChart / BarChart / StackedArea / Heatmap / Gantt ...   │
└────────────────────────▲────────────────────────────────────┘
                         │ 选 hook 取数
┌────────────────────────┴────────────────────────────────────┐
│ 数据层 hooks/metrics/ (Hook 扩展函数)                        │
│  machine_util.js / agv_loaded.js / swap_conflict.js ...     │
└────────────────────────▲────────────────────────────────────┘
                         │ 订阅
┌────────────────────────┴────────────────────────────────────┐
│ 数据源 monitor.js (metricsTimeline / episodeSummary)        │
│  ← SSE /stream/metrics                                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据层：Hook 规范

每个 Hook 是一个纯函数，签名固定：
```js
// hooks/metrics/machine_util.js
export default {
  id: 'machine_util',
  label: '机器利用率',
  unit: '%',
  // 从 monitor store 抽取时序点 [{x, y}, ...]
  series: (monitorStore) => monitorStore.metricsTimeline.map(p => ({
    x: p.step,
    y: (p.metrics?.machine_utilization ?? 0) * 100,
  })),
}
```

**Hook 只做一件事**：从 `monitor.js` 的累积数据里捞需要的字段、做单位换算，输出标准 `{x, y}` 序列。**禁止**包含 ECharts option、CSS、布局逻辑。

新增指标只加一个 Hook 文件 + 注册，不动模板和渲染代码。

> 现有 `MetricsPanel.vue:SIM_METRICS_META` 字典里那 13 个指标就是第一批 Hook 的来源（`machine_utilization` / `agv_loaded_utilization` / `transport_delay_ratio` / `swap_conflict_count` / ...），逐个迁出。

### 2.3 模板层：通用 Chart 组件

把现在 `createMiniOption` / `createBigOption`（`MetricsPanel.vue:360-395`）的耦合拆开。每个 Chart 组件**只接收标准 series 数据**，不知道指标名：

```vue
<!-- components/charts/LineChart.vue -->
<template>
  <v-chart :option="composedOption" autoresize />
</template>
<script setup>
const props = defineProps({
  series: Array,   // [{ name, data:[{x,y}], color? }]
  xLabel: String,
  yLabel: String,
  yMax: Number,
})
// composedOption 只组装 ECharts 样式（线条/坐标轴/legend/tooltip），
// 不写死 "machine_utilization"
</script>
```

**模板清单**（首期）：
- `LineChart` — 时序趋势（取代现在写死的双线折线）
- `BarChart` — 分布对比（取代 machine/agv bar）
- `StackedAreaChart` — 累积视图
- `HeatmapChart` — 占用率（取代现在写死的 occupancy 渲染）
- `GanttChart` — JOB/op 时间轴（与模块一 machine 进度条联动）

热力图特别注意：现有 `MetricsPanel.vue:295-333` 的 occupancy 渲染是纯 div grid，ECharts 原生支持 heatmap，迁到 `HeatmapChart` 后能直接吃 episode 终态的 `heatmaps.occupancy / obstacles`。

### 2.4 配置层：看板配置文件

新建 `config/dashboards/docker_factory.json`（StaticFactory 类似）：
```json
{
  "id": "docker_factory_default",
  "title": "DockerFactory 实时看板",
  "layout": "grid",
  "cards": [
    {
      "hook": "machine_util", "template": "line",
      "xLabel": "step", "yLabel": "%", "yMax": 100
    },
    {
      "hook": "agv_loaded", "template": "line",
      "overlay": ["agv_busy"]
    },
    { "hook": "swap_conflict", "template": "bar" },
    { "hook": "occupancy", "template": "heatmap" }
  ]
}
```

**用户侧能力**（在线自定义）：
- 修改配置文件 → 实时切换看板组合，无需前后端发版
- 前端提供轻量编辑器（拖拽卡片、切模板、绑 Hook）→ 保存到 localStorage 或后端 preset

> 静态优先：先支持 JSON 文件配置驱动渲染，编辑器作为第二阶段。

## 3. 最终形态：离线日志分析（天工版华佗）

### 3.1 与大禹版的关键差异

华佗在大禹那边是**独立系统**，导入的是运维日志（QPS、延迟、报错）。天工版离线分析的对象是**仿真运行日志**：
- 一次仿真 = 一个 episode = 一组 step 序列 + 终态汇总
- 关注的指标是调度/物流维度（makespan、利用率、冲突、队列等待）而不是运维维度

但**底层渲染内核完全复用**——这正是师兄反复强调的"在线离线共用一套 Chart 模板"。

### 3.2 数据流

```
sim_server 跑批 → sky_logs/episode_xxx.jsonl
                         ↓ 用户上传
                  POST /api/logs/upload
                         ↓ 入库（文件目录）
                    run_id = xxx
                         ↓ 触发分析
              POST /api/logs/{run_id}/analyze
                         ↓ 离线 Hook 计算
          metrics_dict = {machine_util:[...], makespan:123, ...}
                         ↓ 前端走同一套模板
                  views/analysis/ + components/charts/
```

### 3.3 离线 Hook（复用在线 Hook 的契约）

离线 Hook 签名和在线一致，唯一区别：数据源从 `monitorStore`（实时）换成 `parsedLog`（解析后的结构化日志）：

```js
// hooks/offline/machine_util.js — 与在线同名 Hook 接口一致
export default {
  id: 'machine_util',
  label: '机器利用率',
  series: (parsedLog) => parsedLog.steps.map(p => ({
    x: p.step,
    y: p.metrics.machine_utilization * 100,
  })),
}
```

→ 前端配置层完全复用（看板 JSON 里只写 `hook: "machine_util"`，由数据源类型决定调在线还是离线版本）。这就是"在线离线共用渲染内核"在天工的落地方式。

### 3.4 日志格式（sim_server 侧需要新增）

sim_server 现在只有 `print()` 调试输出（`sim_server.py:518-568`）。需要新增结构化日志写入：
- 格式：JSON Lines，每行一事件
- 事件类型对齐模块一/二的语义：`machine_start_op` / `machine_idle` / `transfer_started` / `transfer_completed` / `episode_summary`
- 落盘位置：finalpro 的 `sky_logs/episode_{run_id}.jsonl`（已 bind mount）

> 注意：`sim_server._diff_and_emit`（`sim_server.py:302-366`）已经在派生这些事件推给 `/stream/events`，**只需要再加一个 sink 写到 JSONL 文件**，不需要重新实现事件检测逻辑。这是 sim_server 改造的最小切口。

### 3.5 离线交互能力

师兄强调的"离线图表支持时间拖拽、区间筛选"——ECharts 的 `dataZoom` 组件原生支持，模板层统一加上：
```js
dataZoom: [{ type: 'inside', xAxisIndex: 0 }, { type: 'slider' }]
```
在线和离线模板都默认带上，体验一致。

## 5. 设计纪律（迁移的硬约束）

从大禹/华佗经验里继承的三条，后续所有 PR 必须遵守：

1. **任何指标渲染需求，先写 Hook，再选模板，最后改配置**。不允许直接在组件里写死指标名 + ECharts option（现有 `MetricsPanel.vue` 的硬编码模式是反例，要逐步消除）。
2. **新增图表样式 = 加模板组件，不是改老组件**。同一类图表（折线/柱状）所有指标共用。
3. **在线和离线的 Chart 模板必须共用同一份代码**。离线分析页的折线图就是在线看板的那张折线图，数据源不同而已。
